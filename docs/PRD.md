# EasyPlan 项目需求规格说明书 (PRD) v1.2.2

## 1. 项目综述
**项目名称：** EasyPlan
**项目愿景：** EasyPlan 是一个意图驱动的 AI 规划系统，它先识别用户目标类型，再选择合适的拆解策略，把模糊意图转化为可执行、可调整、可持续推进的任务地图。
**核心理念：** 
- 意图驱动：让用户从“盲目忙碌”转向“有意识行动”。
- 行为设计：利用 BJ Fogg 模型，通过 AI 拆解极大降低启动门槛（能力提升）。
- 适度自治：AI 负责规划建议，人类保留最终执行的控制权（HITL）。

## 2. 目标用户与痛点
- **目标用户：** 知识工作者、长期受挫于复杂计划的拖延症患者、追求极致效率的开发者。
- **核心痛点：**
    - 面对大目标时产生的“启动性焦虑”。
    - 传统待办工具录入成本高、维护难。
    - 无法有效地将长期目标分解为即刻可执行的微任务。

## 3. 产品功能矩阵

### 3.1 核心功能 (v1.2.2 已实现)
1. **自然语言意图捕获 (Intent Capture)：**
   - 极简输入框，支持模糊口语输入。
2. **AI 智能拆解引擎 (Agentic Decomposition)：**
   - 基于 LangGraph 的多步推理，将意图拆解为两级扁平化的任务树。
3. **人工确认流与对话式微调 (HITL & Refine)：**
   - 展示 AI 拆解结果，支持用户通过自然语言一键修改或重构计划。
4. **原生任务看板 (Native Task Board)：**
   - 内置“我的一天”和“计划中”视图的闭环手帐，支持跨视图流转与行内快记。
5. **多用户 SaaS 体系：**
   - 基础用户注册/登录、JWT 严格鉴权。
6. **沉浸式交互 (Zen Mode)：**
   - 护眼羊皮纸主题，时间感知问候语，消除空白画布焦虑。

### 3.2 进阶功能 (v1.2.3 Roadmap)
1. **视野控制 (Scope Horizon)：** 
   - 限制宏大目标的拆解深度，仅规划启动阶段。
2. **迷雾解锁 (Fog of War)：** 
   - 阶段完成后动态解锁下一阶段。

### 3.3 交互原则 (v1.2.5 补充)
1. **生成态是单次 run，不是聊天历史：**
   - 每次新生成、下一阶段生成或重试都应清空上一次 reasoning 流；旧 run 如需保留，仅作为可折叠的失败摘要存在。
2. **探索决策先给判断，再给计划：**
   - 对 `exploration_decision` 类型，首屏先给 1-2 句当前判断或建议，再展示澄清、收集、验证、决策的阶段路线。
3. **必须提供逃生口：**
   - AI 生成、重试、下一阶段预览都要允许用户“返回当前计划”或“取消本次生成”，避免用户被锁在生成界面。
4. **时间信息保留但降精度：**
   - 生成界面优先展示时间档位而不是伪精确分钟；进入任务看板后再展示 rounded 预计时长。
5. **全部计划是聚合视图，项目是归属容器：**
   - 同一任务可以同时出现在“全部计划”和某个项目中；“全部计划”用于跨项目总览，不承担独立的 Roadmap 与阶段上下文。

## 4. 技术架构概览 (SaaS 模式)

### 4.1 后端 
- **核心框架：** FastAPI (Python) - 负责异步网关与业务逻辑。
- **Agent 编排：** LangGraph - 全异步执行，带 Checkpoint 持久化。
- **数据结构：** Pydantic V2 - 强制约束 AI 输出结构。
- **存储：** PostgreSQL - 存储用户信息、线程快照。

### 4.2 前端 
- **框架：** React (TypeScript) + Tailwind CSS + Framer Motion。
- **状态管理：** Zustand - 维护全局应用状态机。
- **通信：** Async Queue SSE (Server-Sent Events) 提供真正的断网恢复长连接。

## 5. 关键业务流程
1. **输入阶段：** 用户提交意图字符串 (附带 JWT Auth)。
2. **规划阶段：** Backend 启动 LangGraph 图，Node 1 (Planner) 生成任务树，Node 2 (Validator) 校验。
3. **挂起阶段：** LangGraph 挂起，状态落盘数据库，等待用户信号。
4. **反馈与微调：** Frontend 渲染任务树，用户可提交自然语言要求重拆 (Refine)。
5. **确认阶段：** Backend 接收 Approve 信号，图执行到终态 (Done)。(v1.2.0 将在此处写入原生 Task 表)。

## 6. 非功能性需求
- **响应速度：** AI 拆解过程必须有明确的 SSE 增量日志推送（沉浸感）。
- **生成稳定性：** SSE 断线重连不得重复渲染同一条 reasoning；重试必须开启新 run 视图，不残留上次会话日志。
- **安全性：** 所有核心路由受 JWT 联合 user_id 隔离保护。
- **可落地性：** 界面采用呼吸感的 Zen Mode 美学。
