# EasyPlan v1.3.1 - Execution Engine / Refine Diff

**Release date:** 2026-07-15

EasyPlan v1.3.1 adds a post-commit execution adjustment workflow. When time,
progress, deadlines, or priorities change, users can generate and review a
bounded diff for the current project instead of regenerating the whole plan.

## Highlights

- Three adjustment modes: `time_budget`, `progress_recovery`, and
  `context_change`.
- Four bounded operations: update a task, add a small number of tasks, reorder
  siblings, and change the selected project's My Day projection.
- Before/after preview with one all-or-nothing Apply action.
- Capacity-aware focus selection with deterministic buffer calculation and a
  maximum of five focused parent tasks.
- Completed work, historical phases, Roadmap, long-term loops, reviews,
  checkpoints, Task Assist children, and other projects remain protected.

## Reliability And Safety

- Dedicated durable `execution_refine` runs with run-scoped SSE, heartbeat,
  replay, `snapshot_required` recovery, cancellation, and refresh restoration.
- Canonical scope fingerprints reject stale proposals before mutation.
- Apply is transactional and idempotent: task rows and the committed TaskTree
  are updated together, and repeated Apply returns the stored receipt.
- One active run per project is enforced under concurrency; leases prevent an
  interrupted run from blocking the project indefinitely.
- Task selection and mutation are restricted to the selected project's eligible
  execution scope.

## Frontend

- Added the project-only "调整当前计划" workflow with structured controls and
  grouped diff preview.
- Execution Refine state is isolated from planning and Task Assist state.
- Running and ready proposals recover after refresh using persisted
  thread/request identity and the durable snapshot.
- Apply consumes the authoritative flat receipt, then reloads the project
  snapshot and task view.
- Missing task references use neutral user-facing text; internal IDs are not
  exposed.

## Rollout

Execution Refine remains controlled by feature flags so deployment can enable
the backend and frontend together:

```dotenv
EASYPLAN_EXECUTION_REFINE_ENABLED=true
VITE_EXECUTION_REFINE_ENABLED=true
```

The repository examples keep both flags disabled by default. Deployments should
enable both flags only after the updated backend and frontend are available.

## Compatibility

- Existing plans remain compatible; no historical plan migration is required.
- Existing planning, next-phase, long-term execution, and Task Assist contracts
  remain unchanged.
- The new `execution_refine_runs` table is registered through the project's
  existing SQLAlchemy metadata/startup table-creation flow.
