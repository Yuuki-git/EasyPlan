## API Contract Wording

Replace obsolete draft endpoints and flows with the implemented contract:

```text
POST   /api/intents
GET    /api/threads/{thread_id}
GET    /api/threads/{thread_id}/events
POST   /api/threads/{thread_id}/confirm
POST   /api/threads/{thread_id}/phases/next
DELETE /api/threads/{thread_id}/phases/next/cancel
GET    /api/threads/{thread_id}/phases/next/commit
DELETE /api/threads/{thread_id}
GET    /api/tasks
POST   /api/tasks
PATCH  /api/tasks/{task_id}
DELETE /api/tasks/{task_id}
```

Key contract notes:

- initial, refine, and next-phase runs use unique `request_id` values
- SSE requires `request_id` and carries `thread_id`, `run_type`, and `request_id`
- next-phase generation remains in the same thread
- preview does not replace the committed tree before confirmation
- cancellation requires the matching request ID
- confirmed/SYNCING requests cannot be cancelled
- commit receipt provides deterministic post-confirmation reconciliation
