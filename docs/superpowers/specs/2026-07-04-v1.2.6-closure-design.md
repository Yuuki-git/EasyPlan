# v1.2.6 Portfolio Overview & Answer Layer 收口设计

**状态：** 产品范围已确认，待任务拆分

**目标版本：** `v1.2.6-rc.1`

## 1. 背景

v1.2.6 原计划包含四项能力：

1. 全部计划升级为 Portfolio Overview。
2. `exploration_decision` 先回答问题，再给探索路线。
3. 区分 Retry 与 Regenerate。
4. 降低多次生成后的信息冗余。

其中后三项的大部分能力已在 v1.2.5 的生成体验与下一阶段修复中提前落地。
v1.2.6 不再作为大功能版本展开，而是以小范围收口方式补齐剩余体验并建立明确
验收边界。

## 2. 当前完成度

### 已基本完成

- exploration prompt 强制输出“当前判断 -> 判断依据 -> 下一步探索”。
- Validator 会拒绝只给探索路线、不回答用户问题的结果。
- 前端会把探索 summary 拆成三层展示。
- ERROR 状态使用“重试本次生成”。
- 正常预览调整使用“重新生成”。
- initial、refine、next phase 开启新 run 时会清空旧 reasoning 和 preview。

### 部分完成

- Portfolio Overview 已有独立页面、项目卡片、意图标签和项目跳转。
- 当前卡片没有展示当前阶段、阶段进度和 Next Action。
- stalled 状态的操作文案和行为尚未完全统一。
- 初始规划 `SYNCING` 仍显示本地取消入口，与确认后的不可撤销语义冲突。

### 明确取消

- 不实现“旧 run 折叠为历史摘要”。

当前 run 开始前直接清理旧 reasoning 已经解决信息堆叠问题。保存 run 历史会引入
新的数据结构、恢复规则和视觉负担，不符合本次收口版本的投入收益。

## 3. Portfolio Overview

### 3.1 定位

“全部计划”是项目组合总览，回答三个问题：

1. 我有哪些计划？
2. 每个计划推进到哪里？
3. 下一步最应该做什么？

它不承担项目内任务编辑，也不展示完整 Roadmap 或 Current Phase 面板。

### 3.2 项目卡片

每张项目卡片展示：

- 项目标题
- 可从 `planning_context` 识别时显示 intent 类型标签
- 当前阶段标题
- 当前阶段进度，例如 `2 / 5`
- Next Action 标题
- “进入项目”入口

降级规则：

- 没有 `planning_context` 的手动项目显示“手动计划”。
- 没有 `planning_context` 的 AI 项目显示“直接计划”，不在前端猜测具体 intent。
- 没有当前阶段时显示“尚未建立阶段”。
- 没有 Next Action 时，已完成项目显示“当前阶段已完成”，其他项目显示
  “暂无下一步动作”。
- snapshot 加载失败时保留项目卡片和进入入口，不让单个项目拖垮整个总览。

### 3.3 数据来源

本版本不新增 Portfolio 后端聚合接口。

- 项目列表继续从 planned tasks 聚合。
- thread snapshot 提供 intent、current phase 和 next action。
- 当前阶段进度由当前 `boardTasks` 按 `thread_id + phase_id` 过滤后计算。
- 复用或抽取现有 planning selector，避免在组件中复制阶段统计逻辑。

### 3.4 交互

- 点击卡片进入对应项目。
- 新创建项目保留短暂高亮。
- 加载 snapshot 时卡片保持稳定尺寸，不使用整页 loading 覆盖已有项目。
- “全部计划”不显示项目级 `Roadmap / Current Phase / Next Action` 大面板。

## 4. Exploration Answer Layer

该能力在 v1.2.6 只做回归验收，不再扩展 schema。

必须保持：

- `task_tree.summary` 顺序为“当前判断 -> 判断依据 -> 下一步探索”。
- 当前判断是阶段性判断，不伪装成最终结论。
- UI 先展示回答层，再展示 Action Tree。
- Validator 和 Eval 拒绝缺少回答层或顺序错误的输出。

不新增 `judgment`、`basis` 等 API 字段，继续复用现有 summary 契约。

## 5. Retry、Regenerate 与 Stalled

### 5.1 语义

| 场景 | 操作名称 | 行为 |
| --- | --- | --- |
| 正常预览不满意 | 重新生成 | 开启新的 request，清空当前预览过程 |
| `agent_error` / terminal failure | 重试本次生成 | 开启新的 request |
| SSE 暂时无活动 | 继续等待 | 保持当前 request |
| SSE 连接异常但 run 未失败 | 重新连接 | 重新订阅当前 request，不创建新 request |
| next phase `THINKING/PENDING` | 取消本次生成 | 请求后端取消当前 request |
| `SYNCING` | 返回计划/全部计划 | 只离开等待界面，后台继续提交 |

### 5.2 Stalled 规则

- stalled 只表示超过活动阈值，不代表模型已经失败。
- stalled 状态不得直接创建第二个 planning request。
- “重新连接”复用当前 `activeRun` 和 cursor 恢复机制。
- 后端确认失败后才进入 ERROR，并显示“重试本次生成”。

### 5.3 初始规划 SYNCING

用户确认初始计划后：

- 隐藏“取消本次生成”。
- 新项目提供“返回全部计划”。
- 从既有项目发起的 refine 提供“返回当前计划”。
- 返回只改变当前 UI，不清除 `activeRun`、`threadId` 或 request ID。
- 当前 request 完成后，项目仍应出现在 Portfolio Overview。

next-phase `SYNCING` 继续使用现有“返回当前计划”行为。

## 6. 生成信息降噪

- 新 intent、refine、regenerate 和 next phase 开始时清空旧 reasoning。
- 页面只展示当前 active run 的 reasoning。
- retry/regenerate 不保留或折叠旧 run 记录。
- 当前 committed plan 不属于生成日志，不因新 run 清理而丢失。
- 迟到事件继续由 active run identity 拦截。

## 7. 测试与验收

### Portfolio

- 项目卡片展示当前阶段、进度和 Next Action。
- 手动项目和缺字段 snapshot 使用降级文案。
- 单个 snapshot 失败不影响其他项目。
- 点击项目加载正确 thread。

### Answer Layer

- Prompt、Validator、Eval 和 summary parser 的现有测试继续通过。
- exploration 首屏顺序保持为判断、依据、探索、任务树。

### Retry 与生命周期

- ERROR 只显示“重试本次生成”。
- 正常 PENDING 使用“重新生成”。
- stalled 的“重新连接”不产生新 request ID。
- 初始 `SYNCING` 不显示取消。
- 初始 `SYNCING` 返回全部计划后保留 active run。
- next-phase `SYNCING` 返回当前计划后仍能接收完成事件。
- 每个新 run 开始前 reasoningLogs 为空。

- 不加入下一阶段前的进展补充表单。
- 不修改长期、短期和探索目标的规划模型。
- 不加入 Task Copilot。
- 不进行与本次体验收口无关的视觉重构。
