# EasyPlan v1.2.5 三层规划与阶段视野设计

状态：待用户最终评审
日期：2026-06-23
目标版本：v1.2.5
主验收 Provider：DeepSeek

## 1. 摘要

v1.2.5 将 EasyPlan 从“生成一个可执行任务树”升级为“在长期目标中持续引导当前阶段”。产品原则是：

```text
远期只给地图，近期给计划，眼前给动作。
```

系统首次规划长期成长或探索决策目标时，生成 3–5 个高层 Roadmap 阶段，但只展开当前阶段的任务。当前阶段的 AI Action 全部完成后，用户主动点击“解锁下一阶段”；DeepSeek 生成下一阶段预览，用户确认后才追加到原 thread。

本版本不引入个性化学习、自然语言动态重排或任务级 Copilot。

## 2. 问题定义

v1.2.4 已解决以下问题：

- 根据意图选择正确规划策略。
- 控制长期目标的近期视野。
- 为 Action 提供完成标准、开始提示和降级动作。
- 将计划同步为可编辑、可完成的原生任务。

当前缺口是：

- 长期目标只有当前任务，用户缺少长期方向感。
- 当前阶段完成后，没有可靠的同 thread 续期机制。
- 旧“下一阶段”入口会创建新 intent/thread，破坏计划生命周期。
- TaskBoard 没有明确区分 Roadmap、Current Phase 和 Next Action。

## 3. 目标

1. 为 `long_term_growth` 和 `exploration_decision` 提供 3–5 个高层 Roadmap 阶段。
2. 每次只展开一个 Current Phase 的可执行任务。
3. 确定性地突出一个 Next Action，避免每次完成任务都调用 LLM。
4. 当前阶段完成后，由用户主动触发下一阶段生成。
5. 下一阶段经预览和确认后追加到原 thread，不创建新 thread。
6. 保持 v1.2.4 的 Action Quality、视野控制和任务生命周期能力不回退。
7. 兼容没有 `planning_context` 的旧计划。

## 4. 非目标

v1.2.5 不实现：

- 任务级 AI Copilot，例如“帮我开始”“拆得更细”“缩短到 10 分钟”。
- 计划级 Refine Diff，例如“我今天只有 20 分钟”“我落后了，帮我重排”。
- 根据历史行为自动学习用户偏好。
- 专门的 Phase 数据表、统计报表或阶段分析后台。
- Xiaomi/OpenAI Provider 的专项兼容优化。
- `short_term_delivery` 和 `context_checklist` 的 Roadmap 或阶段续期。
- 从 `exploration_decision` 自动切换为长期执行计划。

对应后续版本：

| 能力 | 目标版本 |
|---|---|
| 任务级一次性微调 | v1.3.0 Task Copilot |
| 计划级动态重排、Refine Diff | v1.3.1 Execution Engine |
| 学习长期偏好并自动个性化 | v1.4 Personalized Planning |

## 5. 已确认产品决策

1. 下一阶段必须由用户主动点击“解锁下一阶段”。
2. 解锁条件仅统计当前阶段的 AI 生成 Action。
3. 用户手动添加的任务不阻塞阶段完成。
4. Roadmap 首次生成 3–5 个阶段。
5. 已完成阶段不可修改；每次解锁只允许调整未开始阶段。
6. 下一阶段采用两步 HITL：生成预览，再确认追加。
7. UI 采用现有 EasyPlan TaskBoard 的适配版 A：顶部轻量 Roadmap、Current Phase 页面标题、克制高亮 Next Action。

## 6. 方案选择

### 6.1 采用方案：扩展 TaskTree + 复用 JSONB

在 `TaskTree` 顶层增加可选 `planning_context`，在现有 `AgentThread.task_tree` JSONB 中保存当前阶段计划和阶段上下文。任务继续存入现有 Task 表，阶段标识写入 Task `metadata`。

优点：

- 不需要数据库迁移。
- 旧 TaskTree 不带新字段时仍然有效。
- 复用现有 ThreadSnapshot、SSE、确认和任务同步链路。
- 保持 Roadmap 与 Task 的语义边界。

### 6.2 未采用：独立 Phase 表

独立表适合阶段统计、查询和复杂协作，但会扩大数据库迁移、事务和 API 范围。v1.2.5 没有这些需求。

### 6.3 未采用：把 Roadmap 编码成 Task Group

该方案实现快，但会把“方向地图”误当成“当前可执行任务”，破坏 Scope Horizon，并让用户能够通过普通任务编辑意外修改路线图。

## 7. 数据契约

### 7.1 TaskTree 扩展

```json
{
  "root": {},
  "summary": "本阶段计划摘要",
  "assumptions": [],
  "planning_context": {
    "schema_version": 1,
    "roadmap": [],
    "current_phase": {},
    "next_action_client_node_id": "phase_01_action_02"
  }
}
```

`planning_context` 为 optional。旧计划缺失时，前端退化为 v1.2.4 TaskBoard。

### 7.2 RoadmapPhase

```json
{
  "phase_id": "phase_01",
  "order": 1,
  "title": "起步准备",
  "objective": "完成资源选择和基础摸底",
  "status": "current"
}
```

约束：

- `phase_id` 在 thread 内唯一；阶段进入 `current` 后必须稳定，仍为 `planned` 时允许随未来路线调整而替换。
- `order` 从 1 开始连续递增。
- `title` 最长 80 字符。
- `objective` 最长 300 字符。
- `status` 仅允许 `planned | current | completed`。
- Roadmap 总阶段数为 3–5。
- 同一时间必须且只能有一个 `current` 阶段，目标完成后可全部为 `completed`。

### 7.3 CurrentPhase

```json
{
  "phase_id": "phase_01",
  "title": "起步准备",
  "objective": "完成资源选择和基础摸底",
  "completion_rule": "all_ai_actions_completed"
}
```

v1.2.5 只支持 `all_ai_actions_completed`。`current_phase` 在正常推进期间必须存在；最终阶段完成后允许为 null，此时 Roadmap 必须全部为 `completed`。

### 7.4 Task metadata

AI 生成任务写入：

```json
{
  "source": "ai",
  "phase_id": "phase_01",
  "phase_order": 1
}
```

手动任务保持：

```json
{
  "source": "manual"
}
```

手动任务不自动获得 `phase_id`，也不参与阶段完成计算。

## 8. 持久化规则

1. `AgentThread.task_tree` 只保存最新 Current Phase 的 TaskTree 和完整 `planning_context`，不累积所有历史阶段任务，避免长期目标突破 TaskTree 节点上限。
2. 历史阶段任务保留在 Task 表，通过 `thread_id + metadata.phase_id` 区分。
3. 每次确认下一阶段后，用新 Current Phase TaskTree 覆盖 `AgentThread.task_tree`。
4. 已完成阶段的信息保留在 `planning_context.roadmap` 中。
5. Task 表现有 `(thread_id, client_node_id)` 唯一约束继续承担重复写入保护。

## 9. Roadmap 生成与更新规则

### 9.1 首次规划

- `long_term_growth`：生成 3–5 个成长阶段，仅展开 Phase 1。
- `exploration_decision`：生成 3–5 个探索阶段，例如澄清、收集、验证、决策，仅展开 Phase 1。
- `short_term_delivery`、`context_checklist`：`planning_context = null`。

### 9.2 下一阶段预览

DeepSeek 输入包括：

- 原始 intent。
- 原始 Intent Profile。
- 当前 Roadmap。
- 当前阶段目标。
- 当前阶段任务及完成状态摘要。
- 用户手动任务只作为上下文，不作为完成门槛。

输出必须：

- 保留所有 `completed` 阶段，字段完全不变。
- 在预览结果中提议将原 `current` 阶段标记为 `completed`，但确认前不得持久化该变更。
- 选择一个新 `current` 阶段。
- 只允许重写原 Roadmap 中 `planned` 的后缀；planned 阶段可调整标题、目标、顺序和 phase_id。
- 只生成新 Current Phase 的任务。
- 继续满足 Action Quality Validator。

### 9.3 确认追加

只有用户确认后才：

- 更新 AgentThread 的 TaskTree 和 planning_context。
- 将新阶段任务同步到同一个 thread。
- 更新 Roadmap 阶段状态。
- 重新计算 Next Action。

取消预览时不修改 Roadmap、任务或 thread 状态。

## 10. 阶段完成与 Next Action

### 10.1 阶段完成

当前阶段满足以下条件时完成：

```text
存在至少一个当前 phase_id 的 AI action
且这些 action 的 status 全部为 completed
```

以下内容不阻塞：

- 手动创建任务。
- Group 节点本身。
- 已删除的任务。
- 其他阶段任务。

如果阶段没有可识别的 AI Action，则视为数据异常，不开放解锁按钮。

### 10.2 Next Action

Next Action 不调用 LLM，按确定性规则计算：

1. 仅选择当前 `phase_id` 下状态为 active 的 AI Action。
2. `depends_on` 中的任务必须全部完成。
3. 按 `sort_order`、创建时间和 id 稳定排序。
4. 选择第一项作为 `next_action_client_node_id`。
5. 阶段完成时值为 null，并显示“解锁下一阶段”。

如果完成的是 Roadmap 最终阶段，后端直接将该阶段标记为 `completed`、将 `current_phase` 设为 null，并显示目标完成状态，不再提供解锁按钮。

任务状态 PATCH 成功后，后端在同一业务事务中重新计算 Next Action。

## 11. API 与工作流

### 11.1 读取

继续使用：

```http
GET /api/threads/{thread_id}
GET /api/tasks?view_bucket=planned
```

ThreadSnapshot 中的 `task_tree` 返回 optional `planning_context`。Task API 继续返回原生任务；前端通过 Task metadata 或响应中的只读 phase 字段完成分组。

### 11.2 启动下一阶段生成

```http
POST /api/threads/{thread_id}/phases/next
```

请求：

```json
{
  "request_id": "uuid"
}
```

响应：

```json
{
  "thread_id": "thread-id",
  "request_id": "uuid",
  "status": "running",
  "events_url": "/api/threads/thread-id/events"
}
```

服务端校验：

- thread 属于当前用户。
- intent 类型支持阶段规划。
- 当前阶段所有 AI Action 已完成。
- 不存在进行中的下一阶段生成。
- `request_id` 未被不同请求语义占用。

### 11.3 预览与确认

复用现有 SSE 和确认接口：

```text
POST phases/next
→ SSE reasoning / task_tree_ready
→ awaiting_confirmation
→ POST /api/threads/{thread_id}/confirm
→ 同 thread append
```

确认按钮文案改为“追加到当前计划”。取消则返回 TaskBoard，不写入任务。

## 12. 幂等与并发

1. 下一阶段请求必须携带 `request_id`。
2. 同一 thread 同时只能有一个 phase generation lease。
3. 相同 `request_id` 重试返回原运行状态或结果，不重复调用 DeepSeek。
4. 不同 `request_id` 在生成中返回 409。
5. 确认接口重复调用不得重复插入任务。
6. Task 唯一约束作为最终防线；冲突时读取已有任务并返回成功状态。
7. 页面快速连点只影响当前 thread 的按钮状态，不阻塞其他计划。

## 13. 前端设计

### 13.1 信息架构

采用已确认的 EasyPlan 适配版 A：

```text
现有 Sidebar
现有 Header
Current Phase 标题与进度
轻量横向 Roadmap
高亮 Next Action
当前阶段任务列表
完成条件 / 解锁下一阶段
添加手动任务
折叠的已完成阶段历史
```

### 13.2 视觉原则

- 保留当前羊皮纸和 Void 主题。
- 不新增第二套侧边栏或阶段导航。
- Roadmap 使用轻量分隔线，不使用大型卡片。
- Next Action 使用克制的语义强调色，不使用紫色动画或营销式渐变。
- 任务卡继续复用现有完成、编辑、删除、My Day 和执行提示交互。
- 卡片和文本在移动端不得溢出或遮挡。

### 13.3 Roadmap 交互

- v1.2.5 Roadmap 默认只读。
- 当前阶段始终突出。
- 未开始阶段只显示标题和目标，不展开任务。
- 已完成阶段任务通过主列表下方的折叠区域查看，不通过 Roadmap 点击切换。
- 移动端 Roadmap 使用横向滚动，当前阶段进入视图时自动对齐。

### 13.4 状态

| 状态 | UI |
|---|---|
| 阶段进行中 | 展示 Next Action 和剩余任务 |
| 阶段完成 | Next Action 消失，显示“解锁下一阶段” |
| 最终阶段完成 | 显示目标完成状态，不显示解锁按钮 |
| 正在生成 | 按钮禁用，展示生成状态 |
| 等待确认 | 进入现有预览界面，按钮为“追加到当前计划” |
| 生成失败 | 保持当前计划不变，显示可重试错误 |
| 旧计划 | 不显示 Roadmap，保持 v1.2.4 布局 |

## 14. 错误处理

| 场景 | 行为 |
|---|---|
| 未完成阶段就请求解锁 | 409，返回未完成 AI Action 数量 |
| 重复生成 | 409 或返回相同 request 状态 |
| DeepSeek 超时/失败 | 不修改计划，释放 lease，允许重试 |
| 输出修改已完成阶段 | Validator 拒绝并触发定向 replan |
| 输出完整长期任务 | Horizon Validator 拒绝 |
| 确认时部分任务已存在 | 幂等合并，不重复创建 |
| 刷新预览页面 | 通过 ThreadSnapshot 和 checkpoint 恢复 |
| 旧 thread 无 planning_context | 隐藏阶段功能，不报错 |

## 15. DeepSeek Prompt 与 Validator

新增 `next_phase` 规划模式：

- 明确只生成一个新 Current Phase。
- 输入并锁定 completed 阶段。
- planned 阶段只允许重写后缀。
- 禁止生成所有未来阶段的任务。
- 保持原始 intent_type，不重新分类为短期交付。
- 新阶段首个 Action 必须可立即开始。
- 所有关键 Action 继续生成 done_criteria、start_hint、fallback_action。

新增校验：

- completed phases immutability。
- exactly one current phase。
- roadmap size 3–5。
- current phase id 与任务 metadata 一致。
- next phase 只包含当前阶段任务。
- same-thread append 不产生新 thread。

### 16.1 后端自动测试

- Roadmap 状态与唯一 current 约束。
- 阶段完成只统计当前阶段 AI Action。
- 手动任务不阻塞。

## 17. 验收场景

### 17.1 长期成长

输入：

```text
我要学习日语 N3，但一直拖着没开始
```

预期：

- 展示 3–5 阶段 Roadmap。
- 只展开起步阶段任务。
- Next Action 明确且可执行。
- 当前阶段 AI Action 全部完成后出现解锁按钮。
- 手动添加任务未完成时仍可解锁。
- 下一阶段确认后追加到同一个 thread。

### 17.2 探索决策

输入：

```text
我想转行产品经理，但不知道怎么开始
```

预期：

- Roadmap 为澄清、信息收集、验证、决策路径。
- 不提前生成长期转行执行任务。
- 下一阶段只根据已完成探索结果调整未来 planned 阶段。

### 17.3 不显示 Roadmap

输入：

```text
今天下午我要写完商业计划书初稿
下班买菜、拿快递，晚上交电费
```

预期：

- 保持 v1.2.4 TaskBoard。
- 不显示阶段进度或下一阶段入口。

## 18. 发布与回滚

1. 先合并 optional schema 和兼容读取。
2. 再启用 DeepSeek 新 Prompt 和 next phase API。
3. 最后启用前端 Roadmap UI。
4. 使用功能开关控制 v1.2.5 UI；关闭时退化为 v1.2.4 TaskBoard。
5. 回滚不得删除 planning_context 或已追加任务，只隐藏新 UI 和禁用 next phase API。

## 19. 实施边界

建议按以下工作包拆分：

1. 后端数据契约与兼容读取。
2. 后端阶段状态、Next Action 和持久化。
3. 后端 next phase 工作流、幂等和 API。
4. DeepSeek Prompt、Validator 和 Eval。
5. 前端 API/store 状态。
6. 前端 Roadmap/Current Phase/Next Action UI。
7. 前端预览、确认、失败恢复和响应式验收。

各工作包的详细文件级计划在本规格通过用户评审后单独生成。
