# EasyPlan v1.2.8 Planning Model Differentiation Design

**Status:** Approved for implementation planning  
**Date:** 2026-07-10  
**Primary provider:** DeepSeek  
**Scope:** `short_term_delivery` and `exploration_decision`

## 1. Summary

EasyPlan 已经完成意图分类、任务质量、三层规划、长期执行循环和稳定的 SSE
生成体验。当前四类意图虽然会进入不同 Prompt，但只有 `long_term_growth`
拥有足够独立的结构化执行模型：`planning_context.schema_version = 2` 可以表达
Roadmap、当前阶段、循环练习、结果检查点和阶段门槛。

v1.2.8 的目标是补齐另外两类仍不成熟的规划模型：

- `short_term_delivery` 从“普通任务树 + 时间盒提示”升级为以交付物、截止约束、
  范围取舍、工作流和缓冲为中心的交付模型。
- `exploration_decision` 从“把当前判断写进 summary 字符串”升级为可校验的判断、
  依据、信息缺口、低成本实验和决策门槛模型。

本版本在 `TaskTree` 中新增可选的 `strategy_context`。它负责表达不同意图的
业务语义；现有 `planning_context` 继续只负责 Roadmap、Current Phase 和 Next
Action。两者职责不得混用。

## 2. Current Output Contract

当前 Planner 的主产出仍是：

```text
IntentProfile -> TaskTree
```

```text
IntentProfile
├── intent_type
├── time_horizon
└── confidence_score

TaskTree
├── root: TaskNode
├── summary
├── assumptions
└── planning_context: PlanningContext | null
```

当前四类 intent 的实际结构差异为：

| Intent | planning_context | 当前成熟度 | 主要缺口 |
| --- | --- | --- | --- |
| `long_term_growth` | schema v2 | 成熟 | 本版本不改 |
| `short_term_delivery` | `null` | 偏弱 | 没有交付物、截止约束、范围取舍和缓冲契约 |
| `context_checklist` | `null` | 足够 | 轻量聚合已符合场景，本版本不改 |
| `exploration_decision` | schema v1 | 中等 | 判断被编码在 summary 文本中，无法稳定校验和展示 |

## 3. Problem

### 3.1 Short-Term Delivery

当前短期交付只要求模型按模块或时间块拆分，并禁止“打开 Word”一类低价值动作。
它能生成可执行任务，但不能稳定回答：

- 最终要交付什么，什么质量算完成；
- 截止时间和可用时间是否被原样保留；
- 哪些内容必须做，哪些内容可以砍；
- 哪些任务位于关键路径；
- 时间估算是否包含合理缓冲；
- 当时间不足时，计划应如何主动收缩。

因此短期目标虽然任务数量与长期目标相近，却没有形成真正的“交付控制模型”。

### 3.2 Exploration Decision

当前探索决策已经要求 `summary` 按“当前判断 -> 判断依据 -> 下一步探索”输出，
但这仍然是脆弱的字符串协议：

- 前端需要解析自然语言标题；
- Validator 只能检查关键词和顺序，不能校验判断本身的结构；
- “分类置信度”和“判断置信度”没有区分；
- 信息缺口、验证实验和决策条件无法建立明确关联；
- Refine 后容易出现 summary 与任务树语义漂移；
- 无法可靠判断用户是否已经具备进入长期执行计划的条件。

## 4. Goals

1. 为短期交付和探索决策建立独立、可校验、可展示的结构化模型。
2. 保持 `TaskNode` 的 Action Quality 契约不变。
3. 保持长期 schema v2、情境清单和旧计划行为不变。
4. 使用确定性 Validator 约束模型，而不是只依赖 Prompt。
5. 让 DeepSeek Eval 可以直接评测交付质量和决策质量。
6. 前端首先展示“交付摘要”或“当前判断”，再展示任务树。
7. 不增加数据库表，不迁移历史 TaskTree。

## 5. Non-Goals

- 不给 `short_term_delivery` 增加 Roadmap 或阶段解锁。
- 不修改 `long_term_growth` schema v2、practice loop 或 phase gate。
- 不修改 `context_checklist` 的轻量聚合策略。
- 不引入 Task Copilot、自动重排、Refine Diff 或个性化记忆。
- 不让 AI 自动决定从探索计划切换到长期执行计划。
- 不生成真实日历排程，不接入外部日历或 MCP。
- 不把模型生成的时间估算伪装成精确承诺。

## 6. Product Decisions

### 6.1 Separate Phase Context from Strategy Context

`planning_context` 和 `strategy_context` 的职责固定如下：

| Intent | planning_context | strategy_context |
| --- | --- | --- |
| `long_term_growth` | schema v2 | `null` |
| `short_term_delivery` | `null` | `delivery` |
| `context_checklist` | `null` | `null` |
| `exploration_decision` | schema v1 | `decision` |

`planning_context` 回答“当前处在哪个阶段”；`strategy_context` 回答“这种目标应该
按什么业务模型执行”。短期交付不因为新增结构而获得 Roadmap。

### 6.2 Delivery Is Organized Around the Artifact

短期交付的第一对象不是任务数量，而是最终交付物。计划必须先定义交付物、质量
底线和截止约束，再定义任务和取舍。

当时间不足时，系统优先砍掉 `can_cut`，其次降低 `should_have`，不得默默删减
`must_have`。

### 6.3 Exploration Gives a Provisional Judgment

探索决策必须先回答用户当前更适合：

- 继续进行低成本验证；
- 暂停并补充条件后再评估；
- 当前暂不建议投入。

该判断是阶段性建议，不是最终结论。系统必须显示判断置信度和仍缺少的信息，避免
把常识性推测包装成确定事实。

### 6.4 User Controls the Transition to Execution

完成探索实验后，EasyPlan 只显示决策门槛是否满足，并邀请用户复盘。即使条件满足，
系统也不能自动把原 thread 改造成 `long_term_growth`。用户确认方向后，应显式创建
新的长期执行计划；自动转换留给后续版本讨论。

### 6.5 Time Remains an Estimate

短期交付允许结构化保存分钟数，以便预算校验，但前端必须遵循现有显示原则：

- 生成过程中使用粗粒度投入档位；
- 确认后的正式交付摘要可显示 rounded 总时长和缓冲；
- 不显示精确到分钟的完成时刻；
- 用户明确给出的时间预算和截止描述必须原样保留。

## 7. Proposed TaskTree Contract

`TaskTree` 新增一个可选字段：

```python
class TaskTree(BaseModel):
    root: TaskNode
    summary: str
    assumptions: list[str]
    planning_context: PlanningContext | None = None
    strategy_context: StrategyContext | None = None
```

`StrategyContext` 使用 discriminator，避免依赖字段猜测类型：

```python
StrategyContext = Annotated[
    DeliveryStrategyContext | DecisionStrategyContext,
    Field(discriminator="strategy_type"),
]
```

### 7.1 Delivery Strategy Context

```python
class DeliverableDefinition(BaseModel):
    title: str
    format: str
    quality_bar: list[str]  # 1-5


class DeadlineDefinition(BaseModel):
    text: str
    is_explicit: bool


class DeliveryTimePlan(BaseModel):
    available_minutes: int | None
    planned_minutes: int
    buffer_minutes: int


class DeliveryScope(BaseModel):
    must_have: list[str]    # 1-6
    should_have: list[str]  # 0-5
    can_cut: list[str]      # 0-5


class DeliveryWorkstream(BaseModel):
    workstream_id: str
    title: str
    output: str
    task_client_node_ids: list[str]  # 1-8


class DeliveryStrategyContext(BaseModel):
    schema_version: Literal[1] = 1
    strategy_type: Literal["delivery"]
    deliverable: DeliverableDefinition
    deadline: DeadlineDefinition
    time_plan: DeliveryTimePlan
    scope: DeliveryScope
    workstreams: list[DeliveryWorkstream]  # 1-5
    critical_path_client_node_ids: list[str]  # 1-8
```


### 7.2 Decision Strategy Context

```python
DecisionDirection = Literal[
    "continue_exploring",
    "pause_and_reassess",
    "not_recommended_now",
]

DecisionConfidence = Literal["low", "medium", "high"]


class CurrentJudgment(BaseModel):
    direction: DecisionDirection
    statement: str
    confidence: DecisionConfidence


class DecisionBasis(BaseModel):
    statement: str
    basis_type: Literal[
        "user_context",
        "known_constraint",
        "working_assumption",
    ]


class DecisionExperiment(BaseModel):
    experiment_id: str
    title: str
    hypothesis: str
    success_signal: str
    effort_level: Literal["low", "medium", "high"]
    task_client_node_ids: list[str]  # 1-6


class DecisionGate(BaseModel):
    review_after: str
    proceed_if: list[str]  # 1-5
    stop_if: list[str]     # 1-5


class DecisionStrategyContext(BaseModel):
    schema_version: Literal[1] = 1
    strategy_type: Literal["decision"]
    question: str
    options: list[str]  # 2-5
    current_judgment: CurrentJudgment
    basis: list[DecisionBasis]  # 1-5
    missing_information: list[str]  # 1-5
    experiments: list[DecisionExperiment]  # 1-3
    decision_gate: DecisionGate
```

Example:


## 8. Deterministic Validation

### 8.1 Shared Rules

- 新计划启用 v1.2.8 时，`strategy_context.strategy_type` 必须与 intent 匹配。
- `long_term_growth` 和 `context_checklist` 的 `strategy_context` 必须为 `null`。
- 旧计划缺少 `strategy_context` 时继续正常解析和展示。
- 所有 `task_client_node_ids` 必须引用当前 TaskTree 中真实存在的 Action。
- 引用 ID 不得重复；不同 workstream 可以共享依赖任务，但必须明确由 Validator 允许。
- 字符串、数组和引用数量必须有 Pydantic 上限，防止 JSON 体积失控。
- 校验失败进入有限次数 replan，feedback 只修复违规字段和相关任务。

### 8.2 Delivery Rules

- `planning_context` 必须为 `null`。
- 用户明确给出的截止描述、可用分钟数和交付格式必须原样保留。
- `time_plan.planned_minutes` 必须等于所有 Action 的 `estimated_minutes` 之和。
- 存在 `available_minutes` 时，`planned_minutes + buffer_minutes` 不得超过可用时间。
- 计划总投入达到 60 分钟时，必须保留非零 buffer；具体比例不做僵硬固定。
- `must_have` 至少一项；`workstreams` 和关键路径必须至少覆盖一个 Action。
- 关键路径 ID 必须是 workstream 已引用的 Action 子集。
- 禁止生成 Roadmap、阶段解锁、低价值破冰和无明确产出的工作流。

### 8.3 Decision Rules

- `planning_context` 必须保留 schema v1 探索路线；不得使用长期 schema v2。
- `current_judgment.statement` 必须直接回答用户问题，不能只说“需要更多信息”。
- `current_judgment` 必须是阶段性建议，不能宣称绝对正确或替用户做最终决定。
- `summary` 继续作为兼容性摘要，但必须与 `current_judgment.statement` 语义一致。
- `missing_information` 至少一项；每个 experiment 必须关联实际 Action。
- `basis_type = working_assumption` 的内容必须明确显示为假设。
- 不得直接生成报名、辞职、投递或持续投入型长期执行计划。
- `decision_gate` 必须同时包含继续条件和停止条件。

## 9. Prompt and Replan Strategy

Planner Prompt 按 intent 注入对应 JSON 子契约和正反例。不要把两个模型同时塞进每次
请求；`short_term_delivery` 只看到 delivery schema，`exploration_decision` 只看到
decision schema。

Replan feedback 使用稳定错误码：

```text
DELIVERY_CONTEXT_MISSING
DELIVERY_EXPLICIT_CONSTRAINT_DRIFT
DELIVERY_TIME_BUDGET_EXCEEDED
DELIVERY_WORKSTREAM_REFERENCE_INVALID
DELIVERY_CRITICAL_PATH_MISSING
DECISION_CONTEXT_MISSING
DECISION_ANSWER_MISSING
DECISION_OVERCONFIDENT
DECISION_EXPERIMENT_REFERENCE_INVALID
DECISION_GATE_INCOMPLETE
```

Repair 不得借机更改 `IntentProfile`、新增短期 Roadmap、重写长期 schema v2，或把
探索计划提前升级为长期执行计划。

## 10. Frontend Experience

### 10.1 Delivery Summary

短期交付 preview 和项目页在任务树上方展示轻量交付摘要：

- 交付物和格式；
- 截止描述；
- 预计执行投入与缓冲；
- Must Have；
- 时间不足时优先砍掉的内容；
- 工作流和关键路径。

它不是 Roadmap，也不能出现“解锁下一阶段”。用户仍可编辑、完成和删除普通任务。

### 10.2 Decision Card

探索决策 preview 在任务树和探索路线之前展示：

- 当前判断；
- 低 / 中 / 高置信度；
- 判断依据；
- 仍缺少的信息；
- 低成本验证；
- 何时复盘、什么条件继续、什么条件停止。

旧计划没有 `strategy_context` 时，前端继续使用当前 summary parser，不得出现空白页。

### 10.3 Portfolio

“全部计划”保持 portfolio overview，不展示完整 strategy context。项目卡片可以显示
一行交付物或当前判断摘要，但这属于 P1 展示增强，不得阻塞核心模型发布。

## 11. Persistence and API

- `strategy_context` 随 TaskTree 写入现有 thread JSONB，不新增数据库列或表。
- initial、refine、preview、confirm、snapshot 和 SSE `plan_ready` 都必须原样携带该字段。
- OpenAPI 和前端 TypeScript 类型必须同步更新。
- task persistence 仍只保存 TaskNode；strategy context 不复制进每条 task metadata。
- Refine 可以更新 strategy context，但必须继续通过 intent-specific Validator。

## 12. Compatibility and Rollout

- `strategy_context` 是 optional；历史 JSON 不回填、不迁移。
- 使用 `EASYPLAN_STRATEGY_CONTEXT_ENABLED` 作为发布开关。
- 开关关闭时保持当前 Prompt 和 Validator 行为。
- 开关开启时，只要求新生成或新 refine 的目标 intent 输出 strategy context。
- DeepSeek 是唯一发布门槛 Provider；其他 Provider 的兼容问题记录为非阻塞项。
- 新 UI 必须支持 legacy fallback，不能要求所有旧项目重新生成。

## 13. Eval Plan

在现有 42 cases 上新增 12 cases，总计 54 cases：

- 6 条 `short_term_delivery`：明确截止、明确时间预算、单一小交付、多模块交付、
  时间不足需要砍范围、用户指定格式与质量标准。
- 6 条 `exploration_decision`：职业转向、是否读研、是否启动副业、方案选择、
  信息不足的购买决策、现实约束下的方向评估。

新增指标：

```text
Strategy Context Coverage
Delivery Contract Pass Rate
Decision Contract Pass Rate
Explicit Constraint Preservation Rate
Strategy Reference Integrity Rate
```

发布门槛：

- 现有 42 cases 不回归；
- DeepSeek 54/54；
- 上述新增指标均为 100%；
- JSON Parse、Intent、Strategy、Horizon、Action Quality 和 Done Criteria 保持 100%；
- case 级失败必须能定位到稳定错误码。

## 14. Risks and Mitigations

### 14.1 JSON Becomes Too Large

限制每个数组数量和字符串长度；strategy context 不复制任务详情，只通过 ID 引用。

### 14.2 Strategy Context Drifts from TaskTree

由确定性 Validator 校验时间总和、Action 引用、关键路径和实验映射。不能只做 schema
parse success。

### 14.3 Decision Advice Appears Too Certain

使用离散置信度、明确 missing information，并禁止自动进入长期执行。

### 14.4 Delivery Model Overplans Small Tasks

允许 1 个 workstream 和 1 个 Action。小型交付仍需定义交付物，但不强制拆成多个模块。

### 14.5 Legacy Plans Render Incorrectly

所有前端 selector 必须先判断 `strategy_context`；为空时走现有 summary 和 TaskTree
展示路径。

## 15. Acceptance Criteria

1. 两种新 strategy context 的 schema、字段上限和 intent 映射被测试锁定。
2. 短期交付可以稳定表达交付物、截止约束、时间预算、范围和关键路径。
3. 探索决策可以稳定表达当前判断、置信度、依据、信息缺口、实验和决策门槛。
4. 所有跨 TaskTree 引用都由 Validator 校验，错误会触发局部 replan。
5. 旧计划、长期 schema v2 和情境清单无行为回归。
6. 前端新计划显示 Delivery Summary 或 Decision Card，旧计划保持兼容。
7. 不新增数据库 schema，不修改 task 生命周期和 SSE envelope。
8. DeepSeek 完整 Eval 达到 54/54，新增与既有核心指标全部达到 100%。
9. Backend、frontend tests、build、lint 和 `git diff --check` 全部通过。
