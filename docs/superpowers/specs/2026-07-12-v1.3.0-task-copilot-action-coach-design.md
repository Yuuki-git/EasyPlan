# EasyPlan v1.3.0 Task Copilot / Action Coach Design

**Status:** Approved for implementation planning

**Date:** 2026-07-12

**Primary provider:** DeepSeek

**Scope:** Task-level execution assistance

## 1. Summary

EasyPlan 已经能够把模糊意图转化为结构化计划，也已经为长期成长、短期交付、
情境清单和探索决策建立了差异化规划模型。v1.3.0 的重点不再是继续扩展整份计划，
而是解决用户真正开始执行某个任务时出现的局部阻力。

本版本引入 Task Copilot / Action Coach。用户从一条未完成任务进入辅助面板，选择：

- `start`：帮我开始；
- `unstick`：我卡住了；
- `decompose`：拆得更细。

系统读取目标任务及其必要的项目上下文，生成结构化、可预览的局部建议。AI 不得
静默修改任务；只有用户确认 Apply 后，后端才执行有限、确定性的任务变更。

## 2. Problem

计划正确并不意味着用户能够顺利执行。用户常见的阻力包括：

- 看得懂任务，但不知道第一个具体动作是什么；
- 已经开始，却被资料不足、范围过大或时间不足卡住；
- 一条任务仍然太大，需要变成可以逐项完成的子任务；
- 当前 `start_hint` 或 `fallback_action` 不适合此刻的真实情况；
- 用户只需要局部帮助，却不得不重新生成或 refine 整份计划。

如果继续复用 plan-level refine，会产生不必要的风险：整棵 TaskTree 被重写、历史
阶段受到影响、生成过程跳离当前任务、以及局部问题被放大成全局规划问题。

## 3. Goals

1. 让用户在项目和“我的一天”中直接对单个任务请求 AI 辅助。
2. 三类 MVP 辅助均返回稳定、可校验的结构化 proposal。
3. 用户确认前不修改任务、TaskTree、阶段状态或 My Day 归属。
4. Apply 只改变目标任务及明确生成的子任务，不重写其他计划内容。
5. 复用现有 SSE envelope、heartbeat、cancel 和错误体验，但隔离 task-assist run。
6. 对 Apply 提供所有权校验、幂等、过期和 stale-task 防线。
7. 保证拆分任务不会让长期阶段进度提前完成或重复计数。
8. 建立独立 Task Assist Eval，不降低现有 54-case Planning Eval。

## 4. Non-Goals

- 不做开放式长期聊天或保存完整对话历史。
- 不自动重写整份计划、Roadmap、strategy context 或 phase review。
- 不自动调整截止时间、任务优先级、其他任务或 My Day。
- 不实现“我今天只有 20 分钟”的全局重排；该能力属于 v1.3.1。
- 不实现 Refine Diff、Resume Prompt 或个性化记忆。
- 不为完成任务、Group 或 practice-loop occurrence 提供 Action Coach。
- 不支持同时对多个任务发起一次辅助请求。
- 不允许 Action Coach 直接创建新项目或切换 intent。
- 不展示真实 chain-of-thought。

## 5. Product Decisions

### 5.1 Entry Point

Action Coach 只出现在可交互、未完成的普通 Action 上：

- 项目任务支持；
- “我的一天”中的同一 task ID 支持；
- AI task 和 manual task 均支持；
- Group、已完成任务、practice-loop occurrence 和只读 preview 不显示入口。

入口使用图标按钮并提供 tooltip“AI 辅助”。点击必须阻止任务卡完成事件冒泡。

### 5.2 Single-Turn Assistance

每次 assist run 是独立单轮请求。用户可以提供一段补充信息，但 v1.3.0 不建立
聊天 transcript。新的请求不会自动携带上一次 assist 的自然语言对话。

### 5.3 Three MVP Modes

#### `start`

目标：提供一个 2–10 分钟内可以立即执行的第一步。

- 结果可直接阅读和执行；
- Apply 的语义是把建议保存为目标任务的 `start_hint`；
- 不创建子任务，不修改标题、完成标准或预计总时长。

#### `unstick`

目标：根据用户描述或当前任务上下文提供 2–3 个恢复选项。

- 必须明确一个推荐选项，但保留用户选择权；
- 每个选项说明行动、预计投入和取舍；
- Apply 需要携带 `selected_option_id`；
- Apply 的语义是把所选行动保存为目标任务的 `fallback_action`。

#### `decompose`

目标：把一条过大的 Action 拆成 2–5 个可执行子任务。

- 先展示完整子任务 preview；
- Apply 后才在原 task 下创建子任务；
- 不删除原 task，不把原 task 改成 Group；
- 原 task 进入 roll-up 模式，其完成状态由 assist children 决定。

### 5.4 Parent Roll-Up Semantics

原任务保持 `node_type = action`，但拥有一个或多个 `source = task_assist` 的直接子任务
后，进入确定性 roll-up：

- 有未完成子任务时，父任务保持 `active` 且不能直接勾选；
- 所有现存 assist children 完成时，父任务在同一事务中自动完成；
- 任一子任务重新打开时，父任务自动恢复 `active`；
- 删除部分子任务后按剩余子任务重新计算；
- 删除最后一个 assist child 后，父任务退出 roll-up 并恢复手动完成；
- 父任务已经完成时禁止再次 decompose；
- 同一父任务已经处于 roll-up 时，v1.3.0 禁止再次 decompose。

长期阶段进度继续只统计原 AI Action。assist children 不单独计入 phase progress，避免
父任务和子任务重复计数；父任务 roll-up 状态变化后再触发现有 next-action 和 phase
progress 重算。

### 5.5 No Silent Mutation

生成 proposal 和 Apply 是两个独立步骤：

```text
idle -> running -> ready -> applied
                    |  \
                    |   -> expired/stale
                    -> cancelled
```

`ready` 之前和用户点击 Apply 之前，业务任务不得变化。

### 5.6 Stay in Context

Action Coach 使用项目内侧面板和移动端 bottom sheet。生成、预览和 Apply 都留在当前
页面。Apply 成功后刷新同一 task ID 的任务视图，不跳转到全局生成页或全部计划。

## 6. Context Boundary

发送给 DeepSeek 的上下文只包含：

- 目标任务：title、description、estimated_minutes、done_criteria、start_hint、
  fallback_action、status；
- 最多两层 ancestor 的 title 和 description；
- 当前项目的 intent_text、TaskTree summary；
- 当前 strategy context 的摘要字段；
- 当前 phase objective 和 next action（如存在）；
- 用户本次补充信息；
- decompose 时已有直接子任务摘要。

禁止发送：

- 其他项目任务；
- 完整 reasoning 历史；
- token、email 或认证信息；
- 无关 phase history；
- 其他用户数据。

## 7. API Contract

### 7.1 Start an Assist Run

```http
POST /api/tasks/{task_id}/assist
```

```json
{
  "request_id": "uuid",
  "mode": "start | unstick | decompose",
  "user_context": "可选，最多 1000 字"
}
```

Response `202 Accepted`:

```json
{
  "task_id": "uuid",
  "thread_id": "thread-id",
  "request_id": "uuid",
  "mode": "start",
  "status": "running",
  "events_url": "/api/tasks/.../assist/.../events"
}
```

Rules:

- `request_id` 是幂等边界；
- 同一 task 同时只允许一个 `running` run；
- 重复提交相同 request 返回同一 run；
- 已完成、Group、practice occurrence 和无所有权 task 返回 404 或 409；
- Provider 固定为 DeepSeek，不接受前端传入模型名称。

### 7.2 Get Run Snapshot

```http
GET /api/tasks/{task_id}/assist/{request_id}
```

返回当前 status、stage、proposal、错误摘要、创建时间和过期时间。刷新或 SSE 重连时，
前端先读取该快照。

### 7.3 Stream Events

```http
GET /api/tasks/{task_id}/assist/{request_id}/events
```

沿用现有 SSE envelope：

```text
event_id
thread_id
request_id
run_type = task_assist
event_type
seq
created_at
payload
```

允许事件：

```text
run_started
task_context_ready
assist_generation_started
assist_validation_started
still_running
assist_ready
done
agent_error
```

事件不得进入 plan-level `activeRun`。前端使用独立的 `useTaskAssist` 生命周期和游标。

### 7.4 Cancel

```http
DELETE /api/tasks/{task_id}/assist/{request_id}
```

仅 `running` 或 `ready` 可取消；重复取消幂等。取消后迟到模型结果不得恢复 proposal。

### 7.5 Apply

```http
POST /api/tasks/{task_id}/assist/{request_id}/apply
```

```json
{
  "selected_option_id": "unstick 模式必填，其他模式为 null"
}
```

Apply response 包含：

- run status；
- 更新后的目标 TaskResponse；
- 新增或更新的 TaskResponse 列表；
- `apply_receipt`，用于重复请求返回同一结果。

## 8. Proposal Contract

所有 proposal 使用 discriminator `proposal_type`。

### 8.1 Shared Task Draft

```python
class AssistTaskDraft(BaseModel):
    draft_id: str
    title: str
    description: str | None
    estimated_minutes: int
    done_criteria: str
    start_hint: str | None
    fallback_action: str | None
```

### 8.2 Start Proposal

```python
class StartAssistProposal(BaseModel):
    schema_version: Literal[1] = 1
    proposal_type: Literal["start"]
    summary: str
    starter_step: AssistTaskDraft
```

Constraints:

- `starter_step.estimated_minutes` 为 2–10；
- 必须是立即可执行动作，不得写“开始做”“准备一下”；
- Apply 只保存其 `start_hint`，不得创建 task。

### 8.3 Unstick Proposal

```python
class RescueOption(BaseModel):
    option_id: str
    title: str
    action: str
    estimated_minutes: int
    tradeoff: str


class UnstickAssistProposal(BaseModel):
    schema_version: Literal[1] = 1
    proposal_type: Literal["unstick"]
    obstacle_summary: str
    recommended_option_id: str
    options: list[RescueOption]  # 2-3
```

Constraints:

- option ID 唯一且 recommended 必须引用真实 option；
- 单个行动建议为 2–20 分钟；
- 不得把局部阻力升级为整份计划重写；
- Apply 把所选 option.action 保存为 `fallback_action`。

### 8.4 Decompose Proposal

```python
class DecomposeAssistProposal(BaseModel):
    schema_version: Literal[1] = 1
    proposal_type: Literal["decompose"]
    summary: str
    completion_rule: Literal["all_subtasks_completed"]
    subtasks: list[AssistTaskDraft]  # 2-5
    dependencies: list[AssistDraftDependency]
```

Constraints:

- draft ID 唯一；dependency 只能引用 proposal 内 draft；
- 不允许环依赖；
- 每个 subtask 必须满足 Action Quality；
- parent <=15 分钟时最多 2 个 subtask；parent <=30 分钟时最多 3 个；否则最多 5 个；
- 不得生成 Group、Roadmap、practice loop 或新的 strategy context；
- Apply 时服务端根据 `request_id + draft_id` 生成稳定 client_node_id。

## 9. Persistence

新增 `task_assist_runs` 表，保存可恢复、可确认的短生命周期 proposal：

```text
id UUID PK
user_id UUID FK users
task_id UUID FK tasks
thread_id string
request_id UUID
mode string
user_context text nullable
status string
stage string nullable
target_task_updated_at timestamp
proposal JSONB nullable
apply_receipt JSONB nullable
error_code string nullable
error_message text nullable
created_at / updated_at / expires_at / applied_at
```

约束：

- unique `(user_id, task_id, request_id)`；
- proposal 默认 24 小时过期；
- error_message 只保存安全摘要，不保存模型原始响应；
- 删除 task 时级联删除 assist runs；
- Apply 前锁定 run 和 task，并比较 `target_task_updated_at`；
- task 已变更时返回 `409 TASK_ASSIST_STALE`，要求重新生成；
- apply receipt 使重复 Apply 返回同一结果。

assist children 继续存入现有 `tasks` 表：

```json
{
  "source": "task_assist",
  "assist_request_id": "uuid",
  "assist_parent_rollup": true,
  "done_criteria": "...",
  "start_hint": "...",
  "fallback_action": "..."
}
```

父任务 metadata 记录 `assist_rollup=true`，不把 proposal 写回 AgentThread.task_tree。

## 10. Transaction and Idempotency

Apply 必须在单一事务中完成：

1. 按 user、task、request 锁定 assist run；
2. 锁定目标 task 并验证 updated_at；
3. 根据 proposal_type 执行允许的 mutation；
4. decompose 时插入 children 和 dependencies；
5. 更新父任务 roll-up metadata；
6. 必要时重算 phase progress 和 next action；
7. 写入 apply receipt 并把 run 标记为 applied；
8. commit 后返回 receipt。

任何一步失败必须整体 rollback。不得出现“run 已 applied 但 children 未插入”的假成功。

## 11. Runtime and SSE Isolation

Task assist 可以复用 AgentRuntime 已验证的 event buffer、heartbeat、run-scoped replay 和
terminal cleanup，但必须扩展为独立 `run_type = task_assist`：

- 不修改 initial/refine/next_phase 的事件语义；
- plan-level `useSSE` 永远忽略 task_assist；
- task panel 关闭时可以取消当前 run；
- 刷新后通过 run snapshot 恢复 ready proposal；
- 内存事件丢失时，以数据库 run snapshot 为事实来源；
- terminal run 和 cancelled tombstone 必须有回收上限。

## 12. Frontend Experience

### 12.1 Panel

桌面端使用右侧 panel，移动端使用 bottom sheet。面板包括：

- 目标任务标题；
- 三段 segmented control；
- 可选补充输入；
- 生成过程摘要；
- proposal preview；
- Apply / 取消 / 重新生成。

### 12.2 Mode-Specific Apply Copy

```text
start     -> 保存为开始提示
unstick   -> 使用这个降级动作
decompose -> 确认拆分
```

只阅读 proposal 不需要 Apply。关闭未应用的 ready proposal 不修改任务。

### 12.3 Roll-Up Task UI

有 assist children 的 Action 显示展开/折叠控制和子任务进度，例如 `2/4`。父任务 checkbox
在存在未完成子任务时禁用并提供 tooltip；子任务全部完成后父任务自动完成。

My Day 中父任务是用户的承诺锚点：

- Assist children 只随已加入 My Day 的父任务隐式返回并嵌套展示；
- child 自身保持 `is_in_my_day=false`，不显示 My Day 按钮，也不能通过 API 独立加入；
- 父任务移出 My Day 后，其隐式 children 同步消失；
- 平铺 API 数据必须在前端按 `parent_task_id` 重建，orphan Assist child 不得成为顶层任务。

### 12.4 Error and Recovery

- 30 秒前不显示网络超时；
- running 可取消；取消成功后清理本地 run 并关闭 panel，取消失败时保留 panel 并显示错误；
- error/stalled 提供“重试本次辅助”和“关闭”；
- stale apply 明确提示“任务已变化，请重新生成建议”；
- 401 拉起现有登录恢复；
- 409 不进行本地假成功；
- 新 request 清理上一次 proposal 和过程日志。

## 13. Validation and Eval

建立独立 `tests/evals/task_assist_cases.jsonl`，首版 18 cases：

- 6 条 start；
- 6 条 unstick；
- 6 条 decompose。

新增指标：

```text
Task Assist JSON Parse Success
Task Assist Mode Match
Task Assist Actionability
Task Assist Scope Compliance
Task Assist Reference Integrity
Task Assist Explicit Constraint Preservation
```

发布门槛：

- DeepSeek Task Assist Eval 18/18；
- 六项指标全部 100%；
- 现有 Planning Eval 54/54；
- Backend、frontend tests、build、lint 和 `git diff --check` 全部通过。

## 14. Security and Privacy

- 所有 task、run、events 和 apply API 绑定 current user；
- 不用 403 区分越权与不存在，统一避免资源枚举；
- user_context 最大 1000 字并进行正常输入校验；
- 前端不能传入任意 patch；Apply 行为由 proposal_type 和服务端规则决定；
- 原始模型错误、Prompt、SQL、token 和 traceback 不进入 API；
- proposal 过期后不能 Apply；
- 每个用户和 task 的并发 run 数受限。

## 15. Rollout

- 使用 `EASYPLAN_TASK_ASSIST_ENABLED=false` 作为后端开关；
- 前端入口还需要 `VITE_TASK_ASSIST_ENABLED`；
- 关闭时不注册入口，API 返回 404 或受控 disabled 响应；
- 先在开发环境启用 DeepSeek，完成 18-case Eval 和手动验收；
- 不迁移旧 task；所有普通未完成 Action 都可以在开关开启后使用；
- v1.2.8 遗留的 strategy lifecycle P2 测试债务在本版本测试阶段补齐，但不改变产品范围。

## 16. Acceptance Criteria

1. 用户可从项目和 My Day 的同一未完成 Action 打开 Action Coach。
2. 三种 mode 使用结构化 proposal，不能退化为自由文本聊天。
3. start 和 unstick Apply 只更新对应 metadata 字段。
4. decompose Apply 原子创建 2–5 个子任务，并启用父任务 roll-up。
5. assist children 不重复计入长期阶段 progress，父任务状态仍能正确驱动 next action。
6. Preview、取消、刷新恢复、过期、stale 和重复 Apply 行为明确且有测试。
7. task-assist SSE 与 plan activeRun 完全隔离，旧 run 不串流。
8. 用户未确认时任何业务任务都不会变化。
9. 所有接口通过 tenant ownership、幂等和错误脱敏检查。
10. DeepSeek Assist Eval 18/18，原 Planning Eval 保持 54/54。
11. My Day 只通过父任务隐式展示 Assist children，child 不能独立加入或成为顶层任务。
