# EasyPlan v1.3.1 Execution Engine / Refine Diff Design

**Status:** Implemented / Released

**Date:** 2026-07-14

**Primary provider:** DeepSeek

**Depends on:** v1.2.5 phase planning, v1.2.7 long-term execution, v1.2.8 strategy context, v1.3.0 Task Copilot

## 1. Product Goal

v1.3.0 solves a local execution problem:

```text
I know what the task is, but I cannot start or continue this one task.
```

v1.3.1 solves a different problem:

```text
The committed plan was reasonable, but reality changed while I was executing it.
```

The version adds a post-commit Execution Engine that turns current execution context into a bounded,
reviewable `PlanDiff`. The user sees exactly what will change and explicitly applies the whole diff.
The system does not regenerate or replace the entire plan.

Core flow:

```text
Committed Project
    -> Execution Context
    -> Refine Diff Proposal
    -> Deterministic Validate
    -> User Review
    -> Transactional Apply
    -> Updated Current Execution View
```

## 2. Product Positioning

EasyPlan now has three distinct adjustment paths:

| User need | Product path | Mutation boundary |
| --- | --- | --- |
| One task is hard to start, blocked, or too large | Task Copilot | One task or its assist children |
| Several current tasks no longer fit reality | Execution Refine | Current project's eligible execution scope |
| The goal or intent itself has changed | New intent / new plan | New planning run |

Execution Refine must not absorb Task Copilot or new-intent behavior.

Routing guidance:

- “这个任务怎么开始” routes to Task Copilot `start`.
- “这个任务卡住了” routes to Task Copilot `unstick`.
- “这个任务太大” routes to Task Copilot `decompose`.
- “我今天只有 20 分钟” routes to Execution Refine `time_budget`.
- “我落后了，帮我重排当前计划” routes to Execution Refine `progress_recovery`.
- “截止日期或优先级变了” routes to Execution Refine `context_change`.
- “我不想做这个目标了” does not auto-refine; ask whether to archive the project or create a new intent.

## 3. MVP Scope

### 3.1 Modes

The MVP supports exactly three modes:

```text
time_budget
progress_recovery
context_change
```

#### `time_budget`

Typical input:

```text
I only have 20 minutes today.
```

Required input:

- `available_minutes`, from 10 to 480.

Behavior:

- choose a realistic focus set from the current project;
- preserve a deterministic buffer;
- use My Day as the immediate execution projection;
- reduce task scope only through an explicit `update_task` operation;
- do not invent a large new branch of work.

Capacity rule:

```text
buffer_minutes = min(20, max(3, ceil(available_minutes * 0.15)))
focus_estimated_minutes <= available_minutes - buffer_minutes
```

There is no fixed required task count. The focus set is capacity-driven and contains at most 5 parent
tasks. If no safe task combination fits, the proposal must say so and suggest a smaller scoped update
instead of pretending the budget is sufficient.

#### `progress_recovery`

Typical input:

```text
I am behind and need to recover the current plan.
```

Behavior:

- read actual completed and active tasks from the database;
- protect completed work and history;
- reorder current work;
- reduce or clarify active task scope when justified;
- add at most 2 missing actions when recovery would otherwise be impossible;
- avoid manufacturing an entirely new phase.

#### `context_change`

Typical input:

```text
The deadline moved forward, and the presentation is now more important than the report.
```

At least one is required:

- `new_deadline`;
- `priority_task_ids`;
- `blocked_task_ids`;
- `user_context` describing the changed constraint.

Behavior:

- preserve the original project goal unless the user explicitly says the goal changed;
- reflect the new constraint in task scope, ordering, and immediate focus;
- ask for clarification when the changed constraint is not actionable.

### 3.2 Explicitly Out of Scope

- Automatic adjustment without user confirmation.
- Selective per-operation Apply.
- Whole-plan TaskTree regeneration.
- Roadmap rewrite or intent conversion.
- Editing completed tasks or completed phases.
- Rewriting historical phase reviews or completion logs.
- Deleting tasks through AI.
- Changing practice-loop definitions, outcome checkpoints, or phase readiness rules.
- Direct mutation of Task Assist children.
- Calendar, MCP, or external task-system sync.
- Personalized adjustment based on long-term user memory.
- Actual-versus-estimated duration learning; this belongs to the next execution-learning slice.

## 4. Eligible Execution Scope

The backend builds the scope deterministically before calling DeepSeek.

### 4.1 Included Context

- selected thread and owner;
- intent profile and strategy-context summary;
- committed current task tree;
- current phase for long-term schema v2;
- active and completed task status needed to understand progress;
- current My Day membership for tasks in this project;
- direct Task Assist roll-up summary, without exposing mutable child operations;
- practice occurrences as fixed capacity context;
- latest successfully applied Execution Refine context for this thread, if one exists;
- user-provided execution context for the current request.

### 4.2 Mutable Tasks

For long-term schema v2:

- only active, incomplete AI actions in the current phase are mutable;
- current-phase Group structure may be used as an insertion or reorder boundary;
- future roadmap phases and historical completed phases are immutable.

For schema v1 projects:

- active, incomplete AI actions in the committed tree are mutable;
- there is no synthetic Roadmap or phase conversion.

Manual tasks:

- may be reordered or have My Day membership changed;
- may not have title, description, estimate, or quality fields rewritten by AI in the MVP.

Protected tasks:

- completed tasks;
- `source=task_assist` children;
- parents with `assist_rollup=true`, except reorder and My Day operations on the parent anchor;
- practice occurrences and loop-owned tasks;
- tasks from other threads or users;
- tasks in completed phases;
- read-only preview tasks.

Protected tasks may be included as context but cannot appear as mutation targets.

## 5. Request Contract

```python
ExecutionRefineMode = Literal[
    "time_budget",
    "progress_recovery",
    "context_change",
]

class ExecutionRefineRequest(BaseModel):
    request_id: UUID
    mode: ExecutionRefineMode
    available_minutes: int | None = Field(default=None, ge=10, le=480)
    new_deadline: datetime | None = None
    priority_task_ids: list[UUID] = Field(default_factory=list, max_length=5)
    blocked_task_ids: list[UUID] = Field(default_factory=list, max_length=5)
    user_context: str | None = Field(default=None, max_length=1000)
```

Deterministic request validation:

- `time_budget` requires `available_minutes`;
- `context_change` requires at least one changed constraint;
- all referenced task IDs must belong to the selected thread and current user;
- protected tasks may be referenced as context but cannot become mutable targets;
- duplicate task IDs are rejected;
- client timezone is not trusted as an arbitrary string; use the authenticated user's existing timezone.

## 6. Refine Diff Contract

### 6.1 Proposal

```python
class ExecutionRefineProposal(BaseModel):
    schema_version: Literal[1] = 1
    proposal_type: Literal["execution_refine"]
    mode: ExecutionRefineMode
    summary: str
    user_facing_reasons: list[str]
    preserved_constraints: list[str]
    operations: list[ExecutionDiffOperation]
    focus_task_ids: list[UUID]
    estimated_focus_minutes: int
    buffer_minutes: int
    warnings: list[str]
```

Limits:

- total operations: 1-12;
- new tasks: at most 3 globally and at most 2 in `progress_recovery`;
- focus tasks: at most 5;
- user-facing reasons: 1-6;
- warnings: 0-5;
- no hidden chain-of-thought or raw model reasoning.

### 6.2 Operation Union

The MVP supports four operation types.

#### Update Task

```python
class ExecutionTaskChanges(BaseModel):
    title: str | None = None
    description: str | None = None
    estimated_minutes: int | None = None
    done_criteria: str | None = None
    start_hint: str | None = None
    fallback_action: str | None = None

class UpdateTaskOperation(BaseModel):
    operation_type: Literal["update_task"]
    task_id: UUID
    changes: ExecutionTaskChanges
    reason: str
```

Rules:

- at least one field must change;
- only mutable AI actions are allowed;
- status, parent, phase, source, client ID, dependencies, and My Day are forbidden here;
- action-quality fields must pass the existing runtime validator;
- an updated estimate must remain positive and plausible for the resulting action.

#### Add Task

```python
class AddTaskOperation(BaseModel):
    operation_type: Literal["add_task"]
    draft_id: str
    parent_task_id: UUID | None
    title: str
    description: str | None
    estimated_minutes: int
    done_criteria: str
    start_hint: str | None
    fallback_action: str | None
    depends_on_refs: list[str]
    insert_after_task_id: UUID | None
    reason: str
```

Rules:

- parent must be an eligible current execution container;
- do not insert under an Assist roll-up parent;
- dependencies may reference eligible existing task IDs or proposal draft IDs;
- dependency graph must remain acyclic;
- the server generates the final globally unique client node ID from request ID plus draft ID;
- the persisted task keeps `source=ai` for phase accounting and adds
  `created_by=execution_refine` plus `execution_refine_request_id` metadata.

#### Reorder Siblings

```python
class ReorderSiblingsOperation(BaseModel):
    operation_type: Literal["reorder_siblings"]
    parent_task_id: UUID | None
    ordered_task_ids: list[UUID]
    reason: str
```

Rules:

- every ID must have the same parent and belong to the eligible current execution scope;
- the operation must list the complete mutable sibling set for that parent;
- protected siblings keep their relative position and are not silently moved across parents;
- reordering does not rewrite dependencies.

#### Set My Day

```python
class SetMyDayOperation(BaseModel):
    operation_type: Literal["set_my_day"]
    task_id: UUID
    is_in_my_day: bool
    reason: str
```

Rules:

- only tasks from the selected thread may be changed;
- Task Assist children cannot be targeted; their visibility follows the parent anchor;
- tasks from other projects currently in My Day are untouched;
- practice occurrences remain user-controlled and are not targeted by Execution Refine;
- `time_budget` focus IDs must correspond to parent tasks set to My Day by the proposal or already in My Day.

For `time_budget`, deterministic capacity includes selected mutable focus plus protected commitments from the
selected project that are already in My Day. If protected commitments already exceed the available budget, the
proposal returns a warning and no fake compliant focus set. For other modes, `buffer_minutes` is `0`.

### 6.3 No Delete Operation

The MVP deliberately has no `delete_task` or `archive_task` operation.

User-facing “do this later” is represented by:

- moving the task lower within its current sibling order;
- removing the current project's task from My Day when the proposal explicitly says so;
- preserving the task in its project and history.

This avoids destructive AI behavior while still reducing immediate execution pressure.

## 7. Deterministic Validation

The LLM proposes; the backend decides whether the proposal is legal.

Validation order:

1. Parse strict discriminated JSON.
2. Verify mode and request constraints.
3. Verify every referenced task and parent belongs to the owner and thread.
4. Reject completed, historical, practice-owned, or Assist-child mutation targets.
5. Validate operation counts and allowed fields.
6. Validate parent, sibling, dependency, and client-ID integrity.
7. Apply the diff to an in-memory copy of task rows and committed TaskTree.
8. Re-run Action Quality and strategy-context integrity checks.
9. Recompute time-budget totals and buffer deterministically.
10. Verify long-term current-phase boundaries and next-action validity.
11. Verify no Roadmap, review, loop, checkpoint, or history field changed.

Invalid model output receives structured replan feedback. Maximum repair attempts remain bounded at 2.
Repair must preserve the mode and user constraints and may only repair invalid operations.

## 8. Persistence and Source-of-Truth Rules

EasyPlan currently has two representations that must remain consistent:

- normalized task rows and dependency rows for execution;
- `AgentThread.task_tree` JSONB for committed planning context.

Apply is one database transaction:

1. lock the owned thread;
2. lock the refine run;
3. lock all affected tasks in stable UUID order;
4. verify the scope fingerprint and target `updated_at` values;
5. update mutable task rows;
6. create new tasks and dependency rows;
7. update sibling `sort_order` values;
8. update My Day flags;
9. mirror AI-task updates, additions, and ordering into `AgentThread.task_tree`;
10. deterministically recompute next action and phase progress;
11. save an immutable Apply receipt;
12. mark the run applied and commit.

Manual tasks that do not exist in TaskTree are changed only in task rows. My Day flags are also task-row-only.

Any failure rolls back the entire diff. There must be no state where the task table contains the new plan but
the committed TaskTree still contains the old one.

## 9. Durable Run Model

Add `execution_refine_runs` through the repository's existing SQLAlchemy metadata and startup table-creation
flow. Do not introduce a new migration framework only for this version. Deployments that manage schema changes
outside the repository must generate equivalent DDL before enabling the feature.

Required fields:

```text
id
user_id
thread_id
request_id
mode
input_context JSONB
scope_snapshot JSONB
scope_fingerprint
status
stage
proposal JSONB
apply_receipt JSONB
error_code
error_message
lease_owner
lease_expires_at
created_at
updated_at
expires_at
applied_at
cancelled_at
```

Constraints:

- unique `(user_id, thread_id, request_id)`;
- PostgreSQL partial unique index: at most one `status=running` run per `(user_id, thread_id)`;
- proposal expiry defaults to 24 hours;
- expired or orphaned leases become `failed/EXECUTION_REFINE_INTERRUPTED` and release the active slot;
- repeated start with the same request ID returns the same run;
- repeated Apply returns the stored receipt;
- safe terminal cleanup is bounded.

Run status:

```text
running -> ready -> applied
   |         |
   +-> cancelled / failed / expired
```

## 10. API Surface

```http
POST   /api/threads/{thread_id}/refine-diffs
GET    /api/threads/{thread_id}/refine-diffs/{request_id}
GET    /api/threads/{thread_id}/refine-diffs/{request_id}/events
DELETE /api/threads/{thread_id}/refine-diffs/{request_id}
POST   /api/threads/{thread_id}/refine-diffs/{request_id}/apply
```

Start returns `202 Accepted`, the durable identity, current status, and events URL.

Apply request is intentionally empty in the MVP except for an optional expected fingerprint:

```json
{
  "expected_scope_fingerprint": "optional fingerprint returned by snapshot"
}
```

The whole validated proposal is applied atomically. Selective operation IDs are not accepted.

## 11. SSE Lifecycle

Add an isolated `run_type=execution_refine`.

Allowed events:

```text
run_started
execution_context_ready
refine_generation_started
refine_validation_started
repair_started
still_running
diff_ready
done
agent_error
snapshot_required
```

Every event uses the existing complete envelope:

```text
event_id
thread_id
request_id
run_type
event_type
seq
created_at
payload
```

Rules:

- event buffer, replay cursor, heartbeat, and cancellation are scoped by thread + run type + request ID;
- no planning or Task Assist stream accepts `execution_refine` events;
- heartbeat stops at ready, cancelled, failed, expired, or applied;
- browser refresh restores the same durable request;
- historical planning `done` and Task Assist events cannot close this run;
- ready is a durable proposal state, not a mutation;
- Apply is a separate authenticated transaction and returns an authoritative receipt.

## 12. Stale and Error Contract

Canonical error codes include:

```text
EXECUTION_REFINE_ACTIVE_RUN
EXECUTION_REFINE_INTERRUPTED
EXECUTION_REFINE_CONTEXT_STALE
EXECUTION_REFINE_INVALID_PROPOSAL
EXECUTION_REFINE_NOT_READY
EXECUTION_REFINE_EXPIRED
EXECUTION_REFINE_NOT_CANCELLABLE
EXECUTION_REFINE_SCOPE_FORBIDDEN
EXECUTION_REFINE_RATE_LIMITED
EXECUTION_REFINE_APPLY_CONFLICT
EXECUTION_REFINE_PROJECT_BUSY
```

The scope fingerprint covers:

- committed TaskTree hash;
- eligible and referenced task IDs;
- task `updated_at`, status, parent, source, phase, sort order, My Day state, and relevant metadata;
- dependency edges;
- current phase ID and finalized review state where applicable.

If any affected fact changes after proposal generation, Apply returns
`EXECUTION_REFINE_CONTEXT_STALE`. The frontend preserves mode and user input and offers “重新生成调整方案”.

Raw provider errors, prompts, SQL, tokens, and tracebacks never reach the client.

## 13. Frontend Experience

### 13.1 Entry

The first release exposes “调整当前计划” only inside a selected project.

It is hidden when:

- there is no committed project;
- the project has an active plan generation, next-phase, Task Assist Apply, or Execution Refine mutation;
- the project has no eligible active tasks;
- the feature flag is off.

The backend enforces the same guard and rejects start while the thread has an active planning/next-phase commit
that could replace the committed execution scope. Frontend visibility is not the security or consistency boundary.

All Plans and My Day may link back to the project entry but do not run cross-project refine in the MVP.

### 13.2 Input Panel

Use an unframed side panel on desktop and a bottom sheet on mobile.

Controls:

- segmented mode control;
- numeric minute input for `time_budget`;
- task pickers for priority or blocked tasks;
- optional deadline input;
- optional context text;
- clear Start and Cancel commands.

Task Copilot remains available on individual tasks and is not embedded in this panel.

### 13.3 Diff Preview

The result page groups operations into simple rows:

```text
Immediate focus
Updated tasks
New tasks
Order changes
My Day changes
Protected constraints
Warnings
```

Each mutation row shows before, after, and a short user-facing reason. Do not display raw task IDs,
JSON, model reasoning, or nested cards.

The MVP has one primary Apply command:

```text
应用本次调整
```

The user applies the entire validated proposal or does not apply it. Partial checkboxes are deferred because
partial acceptance can invalidate capacity and dependency guarantees.

### 13.4 Exit and Recovery

- running may be cancelled; success closes the panel and clears the local run;
- cancellation failure keeps the panel and identity visible with a recoverable error;
- closing a ready proposal does not mutate tasks and allows snapshot recovery before expiry;
- `done` after proposal generation collapses progress and prioritizes the diff;
- successful Apply stays on the current project and refreshes committed snapshot plus current tasks;
- stale Apply preserves input and offers regenerate;
- logout clears all Execution Refine local identity.

## 14. Interaction with Existing Features

### Task Copilot

- Assist children cannot be direct diff targets.
- A roll-up parent is treated as one commitment anchor.
- My Day changes target the parent; children remain implicit.
- Execution Refine cannot dissolve or recreate an assist roll-up.

### Long-Term Execution

- only the current phase is mutable;
- practice loops, occurrence logs, outcome checkpoints, reviews, and readiness are immutable;
- newly added current-phase AI tasks count toward phase progress after Apply;
- next action is recomputed after Apply;
- phase progression still requires the existing user-controlled review decision.

### Strategy Context

- delivery deadline, budget, Must Have, Can Cut, and critical-path constraints are validation inputs;
- decision experiments and decision gates are preserved;
- the diff may adjust execution tasks but cannot replace `strategy_context` in the MVP.

### My Day

- only selected-project parent tasks may be toggled;
- other projects are untouched;
- Assist children remain implicit under their parent;
- My Day stays a virtual execution view, not a second plan hierarchy.

## 15. Security and Privacy

- every route is tenant-scoped by current user and thread;
- missing and foreign resources share non-enumerating 404 behavior;
- prompt context contains only the selected project and bounded execution history;
- user context is limited to 1000 characters;
- one running refine per thread and at most two per user;
- Apply accepts no arbitrary frontend patch;
- operation targets and changes come only from the stored, validated proposal;
- feature flags default off:

```env
EASYPLAN_EXECUTION_REFINE_ENABLED=false
VITE_EXECUTION_REFINE_ENABLED=false
```

## 16. Eval Design

Create `tests/evals/execution_refine_cases.jsonl` with 24 cases:

- 8 `time_budget`;
- 8 `progress_recovery`;
- 8 `context_change`.

Cases must include:

- 10, 20, 45, 90, and 240 minute budgets;
- budgets where no existing task fits;
- completed and historical tasks that must remain unchanged;
- manual tasks;
- Task Assist roll-up parents and children;
- practice occurrences;
- cross-project My Day tasks;
- long-term current phase boundaries;
- short-term delivery constraints;
- exploration decision gates;
- explicit changed deadline and priority;
- invalid references, dependency cycles, stale context, and provider repair.

Metrics:

```text
Execution Refine JSON Parse Success
Execution Refine Mode Match
Execution Refine Reference Integrity
Execution Refine Mutation Scope Compliance
Execution Refine Constraint Preservation
Execution Refine Capacity Compliance
Execution Refine History Immutability
Execution Refine Action Quality
```

Release gate:

```text
Execution Refine Eval: 24/24
All eight Execution Refine metrics: 100%
Planning Eval: 54/54 and all strict metrics 100%

```

The strict CLI must exit non-zero when any required metric is below 100%.

## 17. Manual Acceptance

Use at least these scenarios:

1. A long-term current phase with incomplete one-off tasks: “我今天只有 20 分钟”.
2. A short-term delivery plan: “我落后两天了，但截止日期不变”.
3. A project with an Assist roll-up parent: “今天只保留最关键的工作”.
4. A project with completed tasks and history: “优先级变了，把演示放到最前面”.
5. A project with tasks from another project already in My Day.

Verify:

- the diff appears in the current project, not the global generation page;
- completed and historical work never changes;
- no task is silently deleted;
- time-budget focus plus buffer fits the budget;
- other projects' My Day tasks remain untouched;
- Assist children remain nested and protected;
- Apply updates task rows and committed TaskTree together;
- refresh restores running and ready proposals;
- cancel, stale, expiry, provider error, and repeated Apply have clear recovery;
- closing without Apply leaves the committed project unchanged.

## 18. Future Extensions

Deferred from v1.3.1:

- selective per-operation Apply with dependency-aware revalidation;
- actual-duration capture and estimate calibration;
- AI readiness recommendation based on execution evidence;
- automatic trigger suggestions after repeated delay;
- user preference learning;
- cross-project daily capacity optimization;
- safe archival or explicit task removal operations.
