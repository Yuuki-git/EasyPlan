# EasyPlan v1.2.7-A Long-Term Execution Loop Design

Date: 2026-07-05
Status: Ready for written review

## 1. Summary

`v1.2.7-A` upgrades `long_term_growth` from a short linear task list into a
repeatable execution loop:

```text
One-off Actions
      +
Practice Loops
      +
Outcome Checkpoints
      +
Phase Review
      ↓
Proceed / Extend / Adjust / Override
```

The release introduces weekly practice quotas, durable completion records,
process-and-outcome phase gates, and lightweight phase reviews. It does not
introduce automatic scheduling or AI-controlled phase advancement.

This is the first slice of Planning Model 2.0. Short-term delivery and
exploration model differentiation remain in `v1.2.7-B` and `v1.2.7-C`.

## 2. Problem

The current long-term model treats a phase as complete when all AI-generated
actions are marked completed. This is insufficient for goals such as language
learning, fitness, skill building, and habit formation:

- A small list of one-off tasks does not represent repeated practice.
- Binary task completion does not show consistency or capability growth.
- The next phase receives only a completion count, not execution evidence.
- A user can complete an idealized plan without meeting the phase objective.
- A user who misses the original schedule has no structured way to extend or
  adjust the current phase.

## 3. Goals

- Represent repeated practice without pre-generating future task instances.
- Let each loop define its own weekly quota based on the user's stated capacity.
- Preserve historical completion data across week resets and task deletion.
- Evaluate phase readiness from one-off work, repeated process, and outcome
  evidence.
- Let users proceed, extend, adjust, or override after a lightweight review.
- Keep My Day user-controlled.
- Preserve all current behavior for old plans and non-long-term intents.
- Produce structured data suitable for a future AI phase advisor.

## 4. Non-Goals

- Automatic calendar scheduling.
- Daily or monthly recurrence rules.
- Arbitrary recurrence windows such as every ten days.
- Automatic frequency changes based on behavior history.
- AI-controlled phase advancement.
- Automatic insertion into My Day.
- Long-term personalization across projects.
- Changes to `short_term_delivery`, `context_checklist`, or
  `exploration_decision`.

These belong to the Execution Engine or Personalized Planning roadmap.

## 5. Product Decisions

### 5.1 Advancement Authority

Phase advancement uses a hybrid authority model:

- EasyPlan computes readiness and recommends an action.
- The user makes the final decision.
- A user may override a non-ready recommendation after entering a reason.
- AI may provide advice in a future version but may not advance a phase without
  confirmation.

### 5.2 Recurrence Window

- The first release supports weekly quotas only.
- `target_per_week` is configurable per loop.
- The valid range is 1 through 7.
- A completion counts at most once per user-local calendar day.
- Missing completions do not carry into the following week.
- A week runs Monday 00:00 through the next Monday 00:00 in the loop's stored
  IANA timezone.

No reset job is required. Weekly progress is computed from dated completion
logs and the applicable loop revision.

### 5.3 Duration

- `duration_weeks` is an estimated training window, not a mandatory wait.
- The valid range is 1 through 12.
- The phase can become ready before the estimated end when all required
  conditions are met.
- Reaching the estimated end opens review even when conditions are not met.
- The user can request an early review at any time.

### 5.4 Workload Limits

Prompt defaults guide the model; validators enforce only hard limits.

| Commitment | Prompt default | Hard limit |
|---|---:|---:|
| One-off actions | 1-3 | 0-5 |
| Practice loops | 1 | 0-2 |
| Outcome checkpoints | 1 | 1-2 |
| Total core commitments | 3-5 | 5 |

Groups do not count as core commitments. Each one-off action, practice loop,
and outcome checkpoint counts once. Weekly occurrences do not count as
additional commitments.

Practice loops are optional. They should be generated only when repeated
behavior materially contributes to the objective. Every long-term phase must
have at least one outcome checkpoint.

## 6. Domain Model

### 6.1 One-off Action

An existing EasyPlan task used for setup, research, selection, or a discrete
deliverable. It retains the current task lifecycle and quality fields.

### 6.2 Practice Loop

A durable definition of repeated work:

```json
{
  "loop_id": "n3_vocab",
  "title": "完成一次 N3 词汇练习",
  "target_per_week": 3,
  "duration_weeks": 4,
  "done_criteria": "完成 20 道词汇题并记录错词",
  "timezone": "Asia/Shanghai"
}
```

A loop is not a future task schedule. It owns the rule, progress, and revision
history.

### 6.3 Loop Occurrence

When the user selects "安排到今天", EasyPlan creates one normal Task occurrence
linked to the loop. A loop may have at most one active occurrence.

Completing the occurrence and writing the completion log are one transaction.
After completion, the user may arrange another occurrence, but only the first
valid completion on a local date counts.

### 6.4 Outcome Checkpoint

A required phase result independent from practice volume. Supported evidence
types:

- `numeric`: score, distance, count, or duration.
- `artifact`: link, file reference, or short evidence note.
- `self_assessment`: integer score from 1 through 5.

Each checkpoint defines its evidence type and deterministic pass rule.

### 6.5 Phase Review

A durable record containing:

- one-off action completion;
- loop adherence by week and across the phase;
- outcome evidence;
- user-reported difficulty;
- next-phase time or capacity changes;
- system recommendation;
- user decision;
- override reason when applicable.

A phase can have multiple finalized review records after extension or
adjustment, but only one active draft review.

## 7. Planning Contract

New long-term plans use `PlanningContext.schema_version = 2`. Existing
schema-version-1 plans remain valid.

The current phase adds:

```json
{
  "practice_loops": [
    {
      "loop_id": "n3_vocab",
      "title": "完成一次 N3 词汇练习",
      "target_per_week": 3,
      "duration_weeks": 4,
      "done_criteria": "完成 20 道词汇题并记录错词"
    }
  ],
  "outcome_checkpoints": [
    {
      "checkpoint_id": "vocab_test",
      "title": "完成词汇测试",
      "evidence_type": "numeric",
      "unit": "percent",
      "operator": "gte",
      "target_value": 65
    }
  ],
  "phase_gate": {
    "process_threshold": 0.8,
    "outcome_rule": "all_required"
  }
}
```

For `v1.2.7-A`:

- `process_threshold` is fixed at `0.8`.
- every defined practice loop is required;
- every outcome checkpoint is required;
- required one-off AI actions must be completed;
- no model-generated custom boolean expressions are accepted.

## 8. Persistence

The feature uses first-class relational persistence rather than relying on
Task metadata as the source of truth.

### 8.1 `practice_loops`

Stores ownership, thread, phase, stable loop identity, active status, start and
estimated end, and timezone.

### 8.2 `practice_loop_revisions`

Stores immutable versions of target frequency, title, done criteria, and
effective week.

An adjustment creates a new revision effective from the next Monday. It never
rewrites targets for past weeks.

### 8.3 `practice_loop_logs`

Stores one valid completion record with:

- `user_id`;
- `loop_id`;
- `occurrence_task_id`;
- completion timestamp;
- user-local completion date;
- optional structured completion note.

Unique constraints enforce one valid record per loop per local date and one
record per occurrence task.

### 8.4 `phase_reviews`

Stores draft and finalized reviews, checkpoint evidence, recommendation,
decision, user feedback, and override reason.

### 8.5 Existing `tasks`

Continues to store one-off actions and the currently materialized occurrence.
Occurrence metadata contains the stable loop ID, but logs remain authoritative
for progress.

## 9. Phase Readiness

The backend computes three independent conditions:

```text
one_off_ready
process_ready
outcome_ready
```

### 9.1 One-Off Readiness

All required AI-generated one-off actions in the current phase are completed.

### 9.2 Process Readiness

For each loop:

```text
required_completions =
    ceil(sum(target effective for each planned week) * 0.8)
```

The loop is ready when its valid completion count reaches that value. If a
phase contains no loops, `process_ready` is true.

When targets change, each week is evaluated against the revision effective at
the start of that week. Historical weeks are never recalculated with a new
target.

### 9.3 Outcome Readiness

All required checkpoints have valid evidence satisfying their deterministic
pass rules.

### 9.4 Recommendation

```text
ready      = all three conditions are true
partial    = at least one but not all conditions are true
not_ready  = no condition is true
overridden = user proceeds while recommendation is not ready
```

The review entry opens when the phase is ready, reaches its estimated end, or
the user requests an early review.

## 10. User Flow

### 10.1 Generate and Confirm

The Planner proposes one-off actions, optional loops, required checkpoints,
and the estimated phase duration. The user can change weekly targets before
confirmation.

### 10.2 Execute

The project Current Phase displays:

```text
本阶段任务
循环练习
阶段验证
```

A loop card shows:

```text
N3 词汇练习
本周 2 / 3 次
阶段累计 8 / 12 次
还剩约 2 周
[安排到今天]
```

The button is disabled when:

- an active occurrence already exists;
- the weekly target is met;
- a valid completion already exists for the current local date;
- the loop is paused, completed, or superseded.

### 10.3 My Day

My Day remains user-controlled. It receives only the occurrence explicitly
scheduled by the user. EasyPlan does not pre-create the remaining weekly
quota.

### 10.4 Review

The review pre-fills system facts. The user adds:

- the main difficulty;
- changed availability or constraints;
- optional context for the next phase.

Available decisions:

- proceed;
- extend with the same loop definitions;
- adjust frequency or difficulty;
- override and proceed.

Extension preserves the current revision and increases the estimated window.
Adjustment creates a future-effective loop revision and preserves historical
targets and logs.

### 10.5 Next Phase

The next-phase Planner receives a compact, structured review summary. It does
not receive the complete raw event log.

## 11. Phase Records

Phase history remains inside the selected project.

- Clicking a completed Roadmap phase opens a compact record summary.
- "查看完整复盘" opens the project's Phase Records view.
- The full view shows dates, adherence, evidence, difficulties, decisions, and
  any override.
- All Plans and My Day do not render phase records.
- No new global navigation item is introduced.

## 12. API Responsibilities

The release adds these contracts:

- `GET /api/threads/{thread_id}` returns loop definitions, computed progress,
  checkpoint state, and review availability in the thread snapshot.
- `POST /api/threads/{thread_id}/practice-loops/{loop_id}/schedule-today`
  idempotently creates or returns the active occurrence in My Day.
- Existing `PATCH /api/tasks/{task_id}` completes an occurrence and creates its
  log in the same transaction.
- Existing task deletion removes an uncompleted occurrence without deleting
  the loop or historical logs.
- `PUT /api/threads/{thread_id}/phases/{phase_id}/review` creates or updates the
  active draft review, including checkpoint evidence and user feedback.
- `POST /api/threads/{thread_id}/phases/{phase_id}/review/decision` finalizes
  `proceed`, `extend`, `adjust`, or `override`.
- Existing next-phase generation accepts only a finalized `proceed` or
  `override` review for schema-version-2 plans.

All operations require matching `user_id`, `thread_id`, `phase_id`, and loop
ownership.

## 13. Failure Handling

- Duplicate schedule requests return the same active occurrence.
- Duplicate completion requests do not create duplicate logs.
- Task completion and log creation commit or roll back together.
- A failed request rolls back only the current occurrence.
- Deleting an uncompleted occurrence permits scheduling another occurrence.
- Deleting or hiding a completed task does not delete its log.
- Week changes preserve history and do not carry missing quota.
- Refresh and cross-view reads recompute progress from backend truth.
- Invalid timezone data falls back to the loop's stored timezone, never the
  server timezone.
- A stale phase or loop revision returns a conflict instead of rewriting newer
  state.

## 14. Compatibility and Rollout

- Only new `long_term_growth` plans use schema version 2.
- Existing long-term plans continue using `all_ai_actions_completed`.
- Non-long-term intents remain on their current contracts.
- The feature is guarded by a backend capability flag during rollout.
- API and frontend must tolerate absent loop and review fields.
- Database migrations are additive and must not rewrite existing tasks.

## 15. Security and Privacy

- All loop, log, evidence, and review queries are tenant-scoped.
- Evidence notes must not contain raw model prompts or hidden reasoning.
- API errors must not expose another user's thread or loop existence.
- Structured review summaries sent to the model contain only the minimum data
  required for planning.

## 16. Verification

### 16.1 Backend

- Loop creation and ownership.
- Idempotent schedule-to-today.
- One active occurrence per loop.
- One counted completion per local day.
- Weekly boundaries in at least two timezones.
- No quota carryover.
- Atomic task completion and log creation.
- Revision effective-date behavior.
- One-off, process, and outcome readiness combinations.
- Early review, duration review, extension, adjustment, and override.
- Old-plan and non-long-term compatibility.
- Tenant isolation and conflict handling.

### 16.2 Frontend

- Loop progress on project refresh.
- Schedule-to-today and duplicate-click protection.
- My Day contains only the explicitly scheduled occurrence.
- Completion updates project and My Day consistently.
- Week rollover does not create task backlog.
- Review states and decision actions.
- Roadmap summary and full Phase Records navigation.
- Authentication loss and request-error recovery.

### 16.3 Planning Eval

The DeepSeek suite adds long-term cases that distinguish:

- goals requiring repeated practice;
- milestone-only goals where loops would be artificial;
- realistic weekly targets based on explicit availability;
- output within commitment limits;
- at least one valid outcome checkpoint;
- no daily schedule expansion;
- no future occurrence pre-generation;
- schema-version-2 validity.

The existing 32-case suite remains a non-regression gate.

## 17. Acceptance Criteria

The feature is accepted when:

1. New long-term phases can contain zero to two practice loops.
2. Weekly targets are configurable from 1 through 7.
3. Occurrences are created only after "安排到今天".
4. The same loop has at most one active occurrence.
5. The same local day contributes at most one completion.
6. Missing weekly quota does not carry forward.
7. Progress survives refresh, task deletion, and cross-view navigation.
8. Readiness requires one-off, process, and outcome conditions.
9. Ready phases can enter review before their estimated end.
10. Expired non-ready phases can be extended or adjusted.
11. Override requires a reason and remains visible in history.
12. Completed Roadmap phases expose summaries and full records.
13. Old plans and all non-long-term intents retain existing behavior.
14. Backend, frontend, contract, and DeepSeek gates pass.
