# EasyPlan v1.2.8 Planning Model Differentiation Implementation Plan

> **Execution rule:** Backend and frontend owners implement the tasks below in order. Each task is complete only after its focused tests pass. Reviewer verifies the contract before the release gate.

**Goal:** Give `short_term_delivery` and `exploration_decision` their own structured, validated planning models while preserving long-term schema v2, context checklists, legacy plans, task lifecycle, and SSE behavior.

**Architecture:** Add an optional discriminated `strategy_context` to `TaskTree`. Keep `planning_context` responsible only for phase navigation. Put deterministic cross-field and TaskNode-reference checks in a pure backend service, teach the DeepSeek Planner and Eval runner the two new contracts, and add focused frontend summary components with legacy fallback.

**Tech Stack:** FastAPI, Pydantic V2, LangGraph, DeepSeek, PostgreSQL JSONB, React 18, TypeScript, Zustand, Vitest, React Testing Library.

**Design source:** `docs/superpowers/specs/2026-07-10-v1.2.8-planning-model-differentiation-design.md`

## Scope Guard

In scope:

- `DeliveryStrategyContext` for new `short_term_delivery` plans.
- `DecisionStrategyContext` for new `exploration_decision` plans.
- Deterministic Validator and targeted replan feedback.
- DeepSeek Eval expansion from 42 to 54 cases.
- Delivery Summary and Decision Card in preview and project context.
- Legacy fallback and feature flag rollout.

Out of scope:

- Any change to long-term schema v2, practice loops, outcome checkpoints, phase reviews, or phase gates.
- Roadmap or next-phase generation for `short_term_delivery`.
- Changes to `context_checklist`.
- Database migrations or new tables.
- Task Copilot, automatic rescheduling, Refine Diff, personalization, or calendar integration.
- Automatic conversion from an exploration plan to a long-term plan.
- SSE envelope or active-run lifecycle changes.

## Contract Matrix

| Intent | planning_context | strategy_context |
| --- | --- | --- |
| `long_term_growth` | schema v2 | `null` |
| `short_term_delivery` | `null` | `delivery` |
| `context_checklist` | `null` | `null` |
| `exploration_decision` | schema v1 | `decision` |

This matrix is a release contract. Prompt, Validator, API types, frontend selectors, and Eval must agree with it.

## Manual Product Acceptance

Use at least these inputs:

```text
今天下午四点前完成商业计划书初稿，我只有三个小时
明天中午前交一份 10 页以内的产品分析 PPT
今晚把一封客户道歉邮件发出去
我想转行产品经理，但不知道现在是否适合
我在犹豫要不要读研，担心时间和经济成本
我是否应该开始一个周末副业
```

Verify:

1. Short-term plans lead with the deliverable, preserve the deadline and available time, show a cut line, and never show Roadmap.
2. Small delivery work is not inflated into multiple artificial modules.
3. Decision plans answer first, label confidence, expose missing information, and provide low-cost experiments.
4. Decision plans do not tell the user to resign, enroll, apply, or begin long-term execution immediately.
5. Preview, confirm, refresh, and refine preserve the correct structured context.
6. Legacy projects still open without errors or empty strategy panels.
7. All Plans remains a portfolio overview; detailed strategy content appears only after entering a project.
