# EasyPlan v1.2.7.2 SSE Reliability & Generation UX Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade EasyPlan generation from "black-box waiting plus delayed logs" into real-time, run-scoped, low-noise, recoverable progress feedback.

**Architecture:** Keep SSE as the one-way Agent run transport. Standardize all backend events into a run-scoped envelope, emit product-friendly stage events before and during long-running work, and let the frontend render a compact generation progress panel instead of an unbounded reasoning log. Do not expose model chain-of-thought.

**Tech Stack:** FastAPI, LangGraph, asyncio Queue SSE, Pydantic v2, SQLAlchemy async, React, TypeScript, Zustand, Vitest, Node-based frontend tests, DeepSeek

## Scope Guard

This patch does **not** introduce:

- new planning strategies
- new intent types
- new TaskTree planning fields
- real chain-of-thought display
- WebSocket transport
- another persistent run table unless a reviewer proves the current event buffer cannot support run-scoped replay

The patch is successful when generation feels immediate, calm, and trustworthy: users see progress quickly, old runs never leak into the current run, and completed plans take visual priority over process logs.

## Task 1: Define and Lock the SSE Event Envelope

**Owner:** Backend

**Files:**
- Modify: `app/services/agent_runtime.py`
- Modify: `app/api/schemas.py` if typed API contracts live there
- Test: `tests/test_agent_runtime.py`
- Test: `tests/test_agent_routes_integration.py`

- [ ] **Step 1: Add failing tests for required envelope fields**

Every SSE event sent to the client must contain:

```json
{
  "event_id": "thread-id:initial:request-id:000001",
  "thread_id": "thread-id",
  "request_id": "request-id",
  "run_type": "initial",
  "event_type": "run_started",
  "seq": 1,
  "created_at": "2026-07-08T00:00:00Z",
  "payload": {}
}
```

Required run types:

```text
initial
next_phase
refine
```

Required event types:

```text
run_started
intent_profile_started
intent_profile_completed
strategy_selected
planning_started
validation_started
repair_started
persistence_started
still_running
plan_ready
sync_status
sync_complete
done
agent_error
snapshot_required
```

Tests must verify:

- `plan_ready`, `done`, and `agent_error` include real `request_id`, `run_type`, and `seq`
- `seq` increases within the same `thread_id + run_type + request_id`
- event ids are unique per run
- events from two runs on the same thread do not share a sequence counter

- [ ] **Step 2: Implement a small envelope builder**

Add one backend helper that all SSE send paths use. Do not let individual nodes hand-build partial event bodies.

Rules:

- `event_id` is derived from `thread_id + run_type + request_id + seq`
- `payload` contains event-specific data only
- top-level run identity fields are never omitted
- event payloads must be JSON serializable

- [ ] **Step 3: Migrate existing event senders**

Update all current SSE event sends:

- reasoning/status events
- `plan_ready`
- `sync_status`
- `sync_complete`
- `done`
- `agent_error`
- `snapshot_required`

Acceptance:

- frontend no longer needs to infer run identity
- historical terminal events can be identified and ignored by run

- [ ] **Step 4: Run focused backend tests**

```bash
python -m pytest tests/test_agent_runtime.py tests/test_agent_routes_integration.py -q
```

**Owner:** Backend + Frontend

- Test: `frontend/tests/runEvents.test.mjs`

- [ ] **Step 1: Add backend replay regression tests**

Cover this exact failure mode:

```text
same thread
old initial run has done
new next_phase run has plan_ready
client subscribes to next_phase request without cursor
old initial done must not terminate or pollute the next_phase stream
```

- [ ] **Step 2: Filter replay by run identity**

Replay must be scoped by:

```text
thread_id + run_type + request_id
```

Rules:

- historical `done` from another run cannot end the current stream
- historical `plan_ready` from another run cannot be sent to the current run subscription
- `Last-Event-ID` applies only inside the subscribed run
- `snapshot_required` is emitted only when the subscribed run has a real event gap

- [ ] **Step 3: Keep frontend activeRun strict**

Frontend must accept an event only when all three fields match:

```text
thread_id
request_id
run_type
```

Mismatched events are ignored without changing UI state.

- [ ] **Step 4: Protect old EventSource handlers**

Handlers must check:

```text
eventSourceRef.current === es
activeRun still matches captured run
```

before writing to store.

## Review Checklist

### Backend Must Check

- [ ] Every event uses the new envelope.
- [ ] `request_id` is real, not inferred or hard-coded.
- [ ] `run_type` is correct for initial, refine, and next-phase runs.
- [ ] `seq` is run-scoped and monotonic.
- [ ] Stage events are emitted before slow work, not after all work completes.
- [ ] Heartbeat stops on terminal states and cancellation.
- [ ] Replay never lets an old terminal event terminate a new run.
- [ ] No raw prompt, provider payload, chain-of-thought, secret, or traceback is sent to frontend.

### Frontend Must Check

- [ ] Events are ignored unless `thread_id + run_type + request_id` matches `activeRun`.
- [ ] Old EventSource handlers cannot write store after cleanup.
- [ ] ReasoningStream no longer behaves like an infinite raw log.
- [ ] Success collapses process feedback.
- [ ] Error/stalled keeps process feedback open.
- [ ] Retry/new intent/next phase clears old progress state.
- [ ] Timeout/stalled copy does not appear before 30 seconds.
- [ ] Generated plan remains the main visual focus after `plan_ready` / `done`.

### Product Acceptance

- [ ] The generation process feels alive within 1 second.
- [ ] The UI communicates progress without pretending to show private model reasoning.
- [ ] The user can recover from slow, failed, or disconnected runs.
- [ ] A successful result does not leave noisy process logs in the way.
- [ ] No planning quality metric regresses.
