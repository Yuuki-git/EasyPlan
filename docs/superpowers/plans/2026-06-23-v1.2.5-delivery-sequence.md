# EasyPlan v1.2.5 Delivery Sequence Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Coordinate backend, frontend, DeepSeek evaluation, and release acceptance so v1.2.5 reaches RC without contract drift or unrelated feature work.

**Architecture:** Freeze the additive JSON/API contract first, then run backend state-machine work and frontend pure/UI work in parallel. Integrate only through the checked-in OpenAPI contract and finish with same-thread lifecycle, failure, visual, and DeepSeek gates.

**Tech Stack:** Git, FastAPI/OpenAPI, React/TypeScript, pytest, Vite, DeepSeek API

### Task 3: Run Contract and Lifecycle Integration

**Files:**
- Modify only files implicated by a failing gate; do not broaden scope.

Create one intent of each type and inspect snapshot/task responses:

- [ ] **Step 2: Verify same-thread append identity**

For one completed current phase, record the thread id and task ids before unlock. Generate, preview, and approve the next phase. Confirm:

```text
thread id unchanged
one new current roadmap phase
old current phase becomes completed only after approval
new tasks carry the same thread id and new phase id
old tasks remain queryable
duplicate client-node ids remain zero
```

- [ ] **Step 3: Verify cancellation and recovery**

Generate another preview and cancel it. Confirm committed `AgentThread.task_tree`, roadmap statuses, and task count remain unchanged. Repeat with a browser refresh during preview; the proposal must recover from `interrupt_payload`, then cancel cleanly.

- [ ] **Step 4: Verify completion and deterministic Next Action**

Complete three current-phase AI actions in sequence. After each PATCH, fetch the thread snapshot and confirm `next_action_client_node_id` advances without a new LLM call. Keep a manual task active and confirm it does not block unlock.

- [ ] **Step 5: Verify concurrency and errors**

Send the same request id twice and then a different request id while the first is active. Expected:

```text
same request id -> same running/preview/result, one DeepSeek call
different active request id -> 409
incomplete phase -> 409 with remaining_ai_actions
other user's thread -> 404
DeepSeek failure -> committed plan unchanged and lease released
```
